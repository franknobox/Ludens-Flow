# Ludens-Flow V3 收版前最后补充项

## 1. 文档定位

这份文档用于补充 V3 收版前最后一轮需要完成的能力方向。

它不替代 [ROADMAP.md](/e:/GitDesktop/Ludens-Flow/11_docs/ROADMAP.md) 或 [Ludens_Flow_V3.md](/e:/GitDesktop/Ludens-Flow/11_docs/Ludens_Flow_V3.md)，而是更聚焦于“V3 进入收版前，还差哪几类关键能力需要补齐”。

本文重点不是拆成细碎任务，而是明确最后的补强方向、预期形态和收版判断标准。

## 2. 收版前最后需要补齐的四类能力

### 2.1 Agent 工具调用能力

V3 在 Agent 编排、Prompt、Schemas 和工作流主链上已经具备基础能力，但如果要进一步走向稳定可用的产品形态，Agent 的工具调用能力仍需要补齐。

这一层需要关注：
- 让不同 Agent 具备更明确的可调用工具边界，而不是只依赖自然语言回答
- 建立统一的工具注册、调用、结果返回与异常处理机制
- 让工具调用结果可以被前端、日志和后续节点稳定消费
- 让工具调用和 Prompt / Schema / 状态更新形成一致链路

V3 收版前，至少应达到：
- Agent 可以稳定调用核心工具，而不是停留在“可回答但不可执行”
- 工具调用结果可以进入结构化输出链路
- 失败、超时、无结果等情况有可见反馈

### 2.2 游戏项目文件操作能力

如果 Ludens-Flow 要真正服务游戏开发，尤其是 Unity 项目，就不能只停留在文档和工件层，还需要补齐对项目文件本身的读取、定位和操作能力。

这一层主要包括：
- 对工程内代码文件、配置文件、资源描述文件的读取与定位
- 对 Unity 常见项目文件的基础理解与可控操作
- 为后续编辑器能力接入预留桥梁，例如通过 MCP 接 Unity 编辑器侧能力

当前阶段更适合优先补齐“工程文件级”能力，而不是一开始就依赖完整编辑器控制。

V3 收版前，至少应达到：
- 能稳定查找和读取游戏项目中的关键文件
- 能支撑基础改动、上下文收集和结果返回
- 对 Unity / 游戏工程接入形成可继续扩展的 MCP 接口方向

### 2.3 输入输出相关的后端能力

目前系统已经具备基础工作流与前端界面，但输入输出链路还需要进一步补强，尤其是更贴近真实产品体验的后端能力。

这一层主要包括：
- 流式输出能力，让前端可以看到更连续的生成过程和状态变化
- 多格式文件读取能力，包括文本、代码、配置、项目描述等常见开发文件
- 文件输入与工具调用链路的统一，让“读文件”本身也成为可管理的能力
- 更明确的任务进度、错误状态和处理中反馈

V3 收版前，至少应达到：
- 关键节点支持流式输出或准流式反馈
- 常见开发文件格式可以被稳定读取并送入上下文
- 输入、工具调用、输出结果之间形成更顺的后端闭环

### 2.4 整体体验优化

V3 最后一轮收版，不应只看能力是否“能用”，还需要看整体体验是否已经具备工作台产品的基本完成度。

这部分以前端为主，但不只属于前端。

这一层主要包括：
- 项目、阶段、工件、历史记录之间的关系展示更清晰
- Agent 工作中、工具调用中、失败和完成时的状态反馈更自然
- 对高风险操作、项目管理操作、文件操作提供更明确的确认与回退感
- 让多项目工作台、历史项目、当前项目上下文切换更顺滑

V3 收版前，至少应达到：
- 前端已经不是功能堆叠，而是能清楚支撑项目推进
- 关键操作的反馈足够明确，不依赖用户猜测系统状态
- 项目管理、对话、工件查看、文件能力之间的体验连接更完整

## 3. 这四类能力之间的关系

这四项并不是互相独立的补丁，而是 V3 收版前最后一轮补强的同一条链路：

- Agent 工具调用能力，决定系统是否能从“会说”走向“会做”
- 游戏项目文件操作能力，决定系统是否能真正进入开发现场
- 输入输出相关的后端能力，决定能力链路是否能稳定承接到前端
- 整体体验优化，决定这些能力能否被用户自然地使用和感知

如果只补其中一项，V3 仍然容易停留在“局部能力增强”；只有这四项一起推进，V3 才更接近完整收版形态。

## 4. 收版判断标准

从收版视角看，V3 最后应至少达到以下状态：

1. Agent 不只是能讨论，还能稳定调用一组核心工具完成辅助执行。
2. 系统不只是围绕工件流转，也能进入游戏项目文件本身进行读取、定位和基础操作。
3. 后端不只是返回一次性结果，而能更自然地承接流式反馈、多格式输入和工具结果输出。
4. 前端不只是展示界面，而已经能够作为主要工作台承接项目管理、对话、工件和文件能力。

## 5. 版本展望

如果这一轮补齐顺利完成，V3 的收版形态将不再只是“多 Agent 工作流系统”，而会更接近一个真正面向游戏开发过程的协作工作台：

- 它能理解项目上下文
- 它能进入工程内容
- 它能通过工具执行辅助动作
- 它能以更完整的前后端体验承接这些能力

这也会为后续更深入的 Unity 接入、编辑器侧能力、评测体系和更强的 Agent 协作机制打下更稳定的基础。
