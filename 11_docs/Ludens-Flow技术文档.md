## 1. 项目定位

<p align="center">
  <img src="IMAGE/LF.svg" alt="Ludens-Flow logo" width="120" />
</p>

**Ludens-Flow 是一个面向游戏开发流程的多 Agent 工作台。**

它不是单次对话式的“游戏点子生成器”，也不是只会改代码的 Coding Agent，而是把 AI Agent 放回真实游戏开发流程里：围绕项目、工件、状态、工具、权限和人类决策组织协作，让用户从一个模糊想法逐步推进到策划案、工程方案、评审结果、持续开发指导和引擎工具调用。

项目名称中的 **Ludens** 取自“游戏的人”，强调创作者在游戏开发中的主体性；**Flow** 指结构化工作流，也指创作者被 AI 辅助后更容易进入的开发心流。

### 1.1 核心创新点

| 创新点 | 说明 | 对应评分维度 |
|---|---|---|
| 多 Agent 游戏开发流水线 | Design、PM、Engineering、Review 四类 Agent 分别承担策划、计划、工程和评审职责，避免一个通用 Agent 同时承担所有任务导致上下文污染。 | 创新性、实用价值 |
| 工件驱动而非聊天驱动 | 系统核心输出是 `GDD.md`、`PROJECT_PLAN.md`、`IMPLEMENTATION_PLAN.md`、`REVIEW_REPORT.md` 等可读、可改、可追踪的项目工件。 | 鲁棒性、实用价值 |
| Human-in-the-loop 决策 | Agent 负责流程推进、结构化建议和执行辅助，人负责创意判断、阶段确认、Review 回流和风险选择。 | 人机交互、鲁棒性 |
| MCP 引擎工具层 | 通过统一 Adapter 将 Unity、Godot、Blender 等外部 MCP 工具收敛成 Ludens-Flow 自己的受控能力。 | 创新性、效率 |
| 项目级上下文与权限边界 | 每个项目独立保存状态、工件、工作区、模型路由、Skills 和引擎配置，工具操作必须经过路径与权限约束。 | 鲁棒性、实用价值 |
| Skills 导入与自我沉淀 | 支持外部 Skills 导入、项目级启用，并探索将重复任务流程沉淀为可复用 Skill。 | 创新性、效率 |
| 游戏开发专项能力 | 文案加工台、CSV 导出、GDD 快速导入、AIGC 入口、GitHub 协作视图等能力围绕游戏制作而非泛办公场景设计。 | 实用价值、人机交互 |

### 1.2 与其他游戏开发辅助方案的比较

以下比较关注“用于开发游戏时”的产品定位差异，不等同于对相关产品完整能力的评价。

| 方案 | 典型优势 | 用于游戏开发时的限制 | Ludens-Flow 的差异 |
|---|---|---|---|
| Gemini 等通用对话模型 | 知识面广、讨论成本低、适合快速问答 | 缺少项目状态、阶段边界、工件版本、工具权限和持续流程 | 将对话放入固定项目流，输出可持续维护的游戏开发工件 |
| Claude Code 等 Coding Agent | 工程执行能力强，擅长读写代码、运行命令、处理 diff | 主要面向代码仓库，不天然覆盖游戏策划、文案、关卡、引擎操作和新人学习路径 | 借鉴工程 harness 思路，但扩展到游戏策划、评审、文案和引擎 MCP |
| Gambo 等一键生成型工具 | 快速生成原型，降低试错成本 | 过程较黑盒，用户学习和控制空间有限，难以融入自己的长期项目 | 强调可解释流程、人工决策、项目工件和可回流迭代 |
| TapTap 制造等游戏创作平台 | 面向游戏创作者，入口友好，生态更集中 | 更偏平台化创作和发布，工程/工具链可控性取决于平台能力 | 更偏本地工作台，可连接现有引擎、仓库、文件和外部工具 |
| 传统引擎软件组合 | Unity、Godot、Unreal、Blender 等能力完整，可生产正式项目 | 学习路径陡峭，跨工具协作成本高，新手很难知道下一步做什么 | 在传统工具之上增加 Agent 工作流、上下文管理、权限边界和过程指导 |

## 2. 设计意图

赛道三强调“不只是打游戏，更要造游戏”。Ludens-Flow 的设计目标是让 AI Agent 真正参与游戏制作流程，而不是停留在生成几段设定、几张图片或几行代码。

### 2.1 对游戏开发实践的判断

小型游戏开发往往不是某一个单点任务，而是一组连续决策：

- 这个想法是否能形成最小可玩版本。
- 核心玩法应该先做哪一部分。
- 需要什么工程结构、资源目录和脚本骨架。
- 什么时候该继续推进，什么时候该回到策划或工程阶段修正。
- 生成的文案、关卡、脚本、素材如何进入真实项目。

通用 AI 工具能回答这些问题的一部分，但它通常不知道项目当前处于什么阶段，也不知道哪些内容已经被定稿，哪些文件可以写，哪些工具可以调用。Ludens-Flow 因此把 Agent 放进一个有状态的开发系统里，而不是让它作为一个孤立聊天窗口存在。

### 2.2 Human-in-the-loop 原则

Ludens-Flow 不追求“完全自动做游戏”。对独立开发和学习场景来说，完全自动化反而会削弱创作者的理解和控制。

本项目采用的原则是：

- **流程、指导、执行交给 Agent**：Agent 负责提出结构化问题、整理工件、调用工具、生成候选方案和指出风险。
- **思考、想法、决策交给人**：用户决定游戏方向、玩法取舍、阶段是否定稿、Review 是否回流以及最终方案是否可接受。
- **在使用中学习**：用户通过每一轮 Agent 解释、工件生成、Review 反馈和工具调用过程，理解游戏开发的基本流程。

这使 Ludens-Flow 更像一个“游戏开发教练 + 工程副手 + 项目工作台”，而不是单纯替代用户产出结果的黑盒。

### 2.3 工具整合意图

真实游戏开发需要同时使用多类工具：游戏引擎、代码编辑器、建模软件、版本协作平台、AIGC 网站、文档系统和测试流程。Ludens-Flow 希望把这些工具整合到一个统一入口下：

- 前端工作台负责展示项目状态、工件、对话、工具过程和设置。
- 后端 Agent 系统负责调度流程、构建上下文、调用模型和执行工具。
- `workspace` 负责保存项目工件、用户画像、Skills、配置和工作区边界。
- MCP Adapter 负责连接 Unity、Godot、Blender 等外部创作工具。

用户不需要在每个环节都重新解释项目背景，Agent 也不应该绕过系统权限直接操作本地文件或引擎。

### 2.4 参考项目与吸收点

| 参考方向 | Ludens-Flow 吸收的思路 |
|---|---|
| LangGraph / 状态图思路 | 将复杂任务拆成阶段和状态转移，避免所有逻辑堆在一次对话里。 |
| MetaGPT / 多 Agent 分工 | 将策划、项目管理、工程和评审拆成不同职责，降低上下文污染。 |
| Claude Code / Codex 类工程 Agent | 借鉴权限确认、工具调用、文件 diff、执行过程可见性等工程 harness 思维。 |
| MCP 生态 | 通过协议化工具调用连接外部引擎和创作软件，把 Agent 从“建议者”推进到“受控执行者”。 |
| 游戏开发实践 | 以 GDD、计划、工程方案、Review、持续开发指导为主线，而不是抽象任务管理。 |

## 3. 架构解析

### 3.1 项目主体工程结构

```text
Ludens-Flow/
├─ agent_workbench/
│  ├─ src/ludens_flow/
│  │  ├─ app/                  # FastAPI 入口、API 分组、环境加载
│  │  ├─ core/                 # 状态机、Agent、Schema、项目状态
│  │  └─ capabilities/         # 文件、MCP、Skills、GitHub、文案等能力层
│  ├─ web/                     # React + TypeScript 前端工作台
│  ├─ tests/                   # 后端单元测试与集成测试
│  └─ scripts/                 # 本地启动脚本
├─ workspace/
│  ├─ projects/                # 用户项目状态、工件、配置
│  └─ skills/installed/        # 全局安装 Skills
├─ 11_docs/                    # Roadmap、技术文档、Benchmark、图片资源
└─ 00_meta/                    # repo rules、schema 文档、贡献说明
```

架构核心是把“产品状态”和“Agent 能力”分开：

- `core` 管理阶段、状态、Agent 生命周期和结构化输出。
- `capabilities` 管理 Agent 能调用的外部能力，如文件、MCP、GitHub、文案、Skills。
- `workspace` 保存真实项目数据，避免把项目状态隐含在一次聊天上下文中。
- `web` 提供人机协作界面，让用户看到项目、工件、过程和权限。

### 3.2 技术栈

| 层级 | 技术 | 作用 |
|---|---|---|
| 前端 | React、TypeScript、Vite | Web 工作台、工件视图、设置页、工具过程展示 |
| 后端 | Python、FastAPI | API 服务、SSE 事件、模型调用、状态管理 |
| Agent 内核 | 自研状态机、Agent 基类、结构化 Schema | 管理阶段流转、角色提示词、输出解析和 Review Gate |
| 能力层 | Tool Registry、MCP Adapter、Workspace Access | 管理工具调用、引擎连接、文件边界和权限 |
| 数据层 | Markdown、JSON、项目目录 | 保存 GDD、计划、Review、用户画像、Skills 和配置 |
| 外部协议 | MCP、GitHub API、LLM API | 连接引擎、协作平台和不同模型供应商 |

### 3.3 核心状态机内核

```text
                 ┌────────────────┐
                 │  GDD_DISCUSS   │
                 │  Design Agent  │
                 └───────┬────────┘
                         │ 定稿
                         ▼
                 ┌────────────────┐
                 │   GDD_COMMIT   │
                 │  写入 GDD.md   │
                 └───────┬────────┘
                         ▼
                 ┌────────────────┐
                 │   PM_DISCUSS   │
                 │    PM Agent    │
                 └───────┬────────┘
                         │ 定稿
                         ▼
                 ┌────────────────┐
                 │   ENG_DISCUSS  │
                 │ Engineering    │
                 └───────┬────────┘
                         │ 定稿
                         ▼
                 ┌────────────────┐
                 │     REVIEW     │
                 │  Review Agent  │
                 └───────┬────────┘
                         ▼
             ┌───────────────────────┐
             │ POST_REVIEW_DECISION  │
             │ 人工选择回流或进入开发 │
             └──────┬────────┬───────┘
                    │        │
             回流修正        通过
                    │        ▼
                    │ ┌────────────────┐
                    └─│  DEV_COACHING  │
                      │  持续开发辅导   │
                      └────────────────┘
```

这个状态机解决三个问题：

- **阶段边界**：不同阶段有不同 Agent、上下文、输出目标和工件约束。
- **质量闸门**：Review 阶段不是形式化总结，而是决定是否回流修正的控制点。
- **工件冻结**：进入持续开发后，已定稿工件默认冻结，避免 Agent 随意改写已确认方案。

### 3.4 输入到输出的执行流程

```text
用户输入 / 附件 / 文件
        │
        ▼
输入解析与附件处理
        │
        ▼
读取项目状态、工件、用户画像、Skills、工作区配置
        │
        ▼
根据当前阶段选择 Agent 与 Prompt 片段
        │
        ▼
模型路由选择 Provider Profile / Model / Temperature
        │
        ▼
Agent 生成结构化输出或工具调用意图
        │
        ├── 普通回复：写入对话与 SSE 流式展示
        ├── 工件提交：写入 Markdown / JSON 工件并更新版本
        ├── 工具调用：经过权限、路径和 Adapter 校验后执行
        └── Review 决策：进入人工选择节点
        │
        ▼
前端展示回复、状态、工件、工具过程和下一步操作
```

这条链路的重点是“每一步都有系统边界”。Agent 不直接决定所有事情，必须经过状态机、Schema、工具权限和项目配置。

## 4. 功能介绍

Ludens-Flow 的能力主要集中在 `agent_workbench/src/ludens_flow/capabilities`。这一层不是散落在 Agent 里的临时函数，而是系统能力边界。

```text
capabilities/
├─ artifacts/      # 工件读取、保存、版本化
├─ context/        # 用户画像、Prompt 模板、上下文注入
├─ copywriting/    # 游戏文案生成与导出
├─ github/         # GitHub 只读协作信息
├─ ingest/         # 图片、文件、附件解析
├─ mcp/            # Unity/Godot/Blender/Unreal MCP Adapter
├─ skills/         # Skills 导入、安装、项目级启用
├─ tools/          # Agent 可调用工具注册
└─ workspaces/     # 工作区路径审核与权限
```

### 4.1 能力总览

| 能力 | 当前作用 | 用户价值 |
|---|---|---|
| Artifacts | 管理 GDD、计划、Review、Notes 等工件 | 让开发结果从聊天记录变成可维护资产 |
| Ingest | 解析图片、文本、代码、PDF 等输入 | 用户可以把已有资料直接带入项目 |
| Context | 注入用户画像、项目工件、Skills、引擎信息 | Agent 回答更贴合当前项目，而不是泛泛回答 |
| Workspaces | 限制 Agent 只能读写用户授权目录 | 降低本地文件误操作风险 |
| Tools | 注册搜索、文件、工程读写、项目操作等工具 | 让 Agent 从“建议”进入“受控执行” |
| MCP | 将外部引擎 MCP 工具映射为统一能力 | 让 Agent 可以理解和操作游戏引擎环境 |
| Copywriting | 面向游戏文案生成台词、描述、CSV | 补齐游戏开发中大量文字生产需求 |
| GitHub | 只读获取仓库协作信息 | 让项目节奏、PR、Issue、提交更可见 |
| Skills | 导入外部 Skill、自我沉淀重复流程 | 让 Agent 能力可扩展、可复用 |

### 4.2 MCP 引擎能力

Ludens-Flow 不把外部 MCP 工具直接暴露给 Agent，而是先映射成自己的能力层。例如：

| Ludens-Flow 能力 | Unity | Godot | Blender | 说明 |
|---|---|---|---|---|
| `engine.list_scene` | 读取 Hierarchy | 读取 Scene Tree | 读取场景对象 | 理解当前场景结构 |
| `engine.create_object` | 创建 GameObject / Prefab | 创建 Node | 创建 Mesh / Object | 生成或放置对象 |
| `engine.move_object` | 修改 Transform | 修改 Node Transform | 修改 Object Transform | 调整位置、旋转、缩放 |
| `engine.save_scene` | 保存 Scene | 保存 Scene | 保存 `.blend` | 保存当前工作结果 |
| `engine.create_script` | 创建 C# 脚本 | 创建 GDScript | 执行/生成 Blender Python | 生成工程脚本 |
| `engine.read_console` | 读取 Console | 读取日志 | 读取执行结果 | 辅助定位错误 |

这种设计避免 Agent 直接依赖某个 MCP 仓库的工具名。底层 MCP 可以变化，但上层产品能力保持稳定。

### 4.3 文案能力

文案能力归属于 Design Agent 的子能力，不作为独立 Agent 存在。原因是游戏文案需要理解玩法设定、世界观、角色定位和工件上下文。

当前文案工作台支持：

- 选择文案类型，例如角色台词、物品描述、任务文本、世界观片段。
- 填写用途、数量、长度、写作风格、必须包含和必须避免。
- 添加外部参考文件。
- 生成候选文案，并支持单条复制、Markdown 导出、CSV 批量导出。
- 对台词类内容生成更接近游戏编辑器可用的表格结构。

### 4.4 Skills 能力

Skills 是 Ludens-Flow 的能力扩展机制：

- 全局安装在 `workspace/skills/installed/`。
- 每个项目独立启用，不影响其他项目。
- Skill 通常包含 `skill.json`、`prompt.md`，可选包含 assets 和 examples。
- Agent 调用前会根据当前项目、当前 Agent 类型和启用状态加载 Skill Context。
- 自我沉淀方向用于把用户反复使用的任务流程整理成可复用 Skill。

## 5. 人机协作流程

下面模拟一个新人使用 Ludens-Flow 开发小型游戏 Demo 的完整流程。

### 5.1 从想法到 GDD

用户只有一个粗略想法，例如“做一个以校园为背景的塔防游戏”。Design Agent 不直接替用户下结论，而是通过讨论澄清：

- 核心玩法是什么。
- 目标平台和操作方式是什么。
- 美术风格和制作成本是否匹配。
- 最小可玩版本应该包含哪些内容。
- 当前项目适合 Unity、Godot 还是通用 Demo 视角。

讨论收敛后，系统生成 `GDD.md`。用户可以人工编辑，也可以导入已有 GDD 文档进入快速开发流程。

### 5.2 从 GDD 到项目计划

PM Agent 读取 GDD 后，生成 `PROJECT_PLAN.md`：

- 拆分 MVP 范围。
- 标记核心风险。
- 区分必须做、可以延后和不建议当前做的内容。
- 给出阶段性任务顺序。

这一步的价值不是替用户做项目管理表格，而是让新人理解“先做什么、为什么先做、哪些内容现在不该做”。

### 5.3 从计划到工程方案

Engineering Agent 读取 GDD 和 Project Plan 后，生成 `IMPLEMENTATION_PLAN.md`：

- 推荐工程目录。
- 设计核心脚本、数据结构、场景结构和调试路径。
- 根据项目引擎字段注入 Unity / Godot / Unreal / Generic 的专用提示片段。
- 在 MCP 模式开启时，可以进一步读取场景、创建脚本、写入文件或调用引擎工具。

### 5.4 Review 与人工决策

Review Agent 会审查前三个阶段的工件一致性：

- GDD 是否过大或过虚。
- PM 计划是否覆盖关键风险。
- Engineering 方案是否与目标引擎和 MVP 范围匹配。
- 是否存在应该回流修正的问题。

Review 完成后，用户可以选择回到某一阶段修正，或进入持续开发辅导。这里是典型的 Human-in-the-loop 节点：AI 提供评审意见，人决定是否接受和如何推进。

### 5.5 持续开发与工具协作

进入 `DEV_COACHING` 后，Ludens-Flow 不再频繁改写核心工件，而是围绕实际开发辅助用户：

- 读取项目工作区文件。
- 生成或 patch 脚本。
- 创建目录和资源文件。
- 调用 Unity / Godot / Blender MCP 工具。
- 展示工具执行过程和权限确认。
- 用文案工作台生成台词 CSV。
- 通过 GitHub 页面查看协作状态。
- 将反复出现的工作流程沉淀为 Skill。

最终用户获得的不只是一个 AI 回答，而是一组可继续迭代的项目资产：工件、代码、配置、文案、Skills 和引擎侧结果。

---

## 6. 未来规划

Ludens-Flow 的长期目标不是只做一个比赛 Demo，而是探索“AI 如何真正进入游戏开发过程”。

### 6.1 产品与工程方向

- **更完整的工程执行闭环**：补齐任务执行循环、Patch/Diff 体验、命令执行、测试反馈和权限确认。
- **更多引擎兼容**：继续推进 Unity、Godot、Blender、Unreal 的 MCP 接入，让不同引擎共享统一能力层。
- **更成熟的 Skills 生态**：完善外部导入、自我沉淀、项目级启用、适用阶段、工具权限和输出 Schema。
- **游戏内模型能力接入**：让开发者把工作台中配置好的 LLM 能力接入自己的游戏，用于 NPC、剧情、任务提示等运行时场景。
- **本地桌面应用化**：在 V3 稳定后评估 Tauri 打包，降低用户启动后端和前端的门槛。
- **外部 AIGC 服务整合**：让图片、声音、3D 模型、原型设计等外部服务更自然地回流到项目工作台。

### 6.2 AI 开发游戏 Benchmark

在产品之外，我们希望建立面向“AI 辅助开发游戏”的 Benchmark。原因是现有很多评测更关注通用问答、代码题或单次工具调用，难以衡量一个 Agent 是否真的能帮助用户完成游戏开发流程。

计划中的 Benchmark 可以覆盖：

- 需求澄清能力：能否把模糊游戏想法收敛成可执行 GDD。
- 工件一致性：GDD、计划、工程方案是否相互匹配。
- 工具使用质量：是否能正确读取文件、调用引擎、处理失败并继续修正。
- 人机协作体验：用户是否容易理解下一步，是否能掌握决策权。
- 鲁棒性：面对附件错误、模型失败、工具失败、阶段回流时是否能稳定恢复。
- 真实产出：是否能生成可导入游戏工程的代码、CSV、资源配置和引擎侧对象。

这也对应比赛评分中的技术维度和应用维度：不仅评估“AI 能不能说”，更评估“AI 能不能在真实开发系统里稳定协作”。

---

## 结语

Ludens-Flow 的核心观点是：游戏制作 Agent 不应该只是“生成游戏内容”的工具，而应该成为一个有边界、有状态、有工件、有工具能力的协作系统。

它把游戏开发拆解为可理解、可审阅、可回流、可执行的流程，让新人开发者能够在 AI 辅助下完成从想法到 Demo 的学习与创作闭环。
