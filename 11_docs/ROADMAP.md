# Ludens-Flow 后续开发计划 (Roadmap)

## V3 收版必做清单

> 以下为 V3 收版前的硬门槛能力，优先级高于扩展性探索项。

1. **工程化入口收口（必须）**
   - 完成 `pyproject.toml` 与标准入口命令（`ludensflow` / `ludensflow-api`）。
   - 去除 `sys.path` 注入依赖，统一包内入口与导入路径。
   - 增加安装与启动 smoke test，保证本地/CI 可复现。
   - 进度：已完成（`216b570`）。

2. **Workflow Core 稳定性门禁（必须）**
   - 统一节点失败、解析失败、超时等异常恢复策略。
   - 补齐主链路端到端回归（多轮、回流、冻结/解冻、并发）。
   - 增加误跳转/误回退等关键断言，避免流程漂移。
   - 进度：已完成（`6965419`）。

3. **状态与项目生命周期完整性（必须）**
   - 为状态与项目元数据建立 `schema_version` 与 migration 机制。
   - 提供项目导出/导入能力（至少覆盖 state + artifacts）。
   - 补齐 reset/migrate 关键操作审计日志。
   - 进度：已完成（`6965419`）。

4. **前端实时同步最小闭环（建议按必须执行）**
   - 增加 SSE 推送通道，支持生成中状态与增量刷新。
   - 降低“请求-整包刷新”依赖，提升实时反馈一致性与丝滑度。
   - 进度：已完成（SSE 事件链路 + 前端增量渲染 + 项目切换/新建去阻塞整包刷新）。

5. **Unity 只读能力安全边界（必须）**
   - 强化路径沙箱、扩展名白名单、大小与编码上限。
   - 统一 Unity 工具错误返回结构，减少前后端分支复杂度。
   - 进度：部分完成（路径沙箱与统一错误结构已完成；扩展名白名单、大小与编码上限按当前产品策略暂缓）。

6. **收版验收基线（必须）**
   - 建立固定 smoke/benchmark 套件（CLI + API + Web）。
   - 固定核心指标：成功率、时延、解析失败率、误回流率。
   - 提供可勾选的 Release Checklist，作为收版闸门。
   - 进度：未完成（保留 smoke 基础能力，benchmark/指标化/checklist 待补齐）。

## 1. 常态化的稳定性、体验性优化
- **核心目标**：持续提升系统稳定性、状态流转一致性与整体可维护性，优先解决影响体验和可扩展性的基础问题。
- **具体方向**：
  - 继续打磨多模态与附件链路的边界行为，重点关注失败反馈、大小与类型限制、开发态与产品态一致性。
  - 完成文件写入能力接入后的权限稳定性，重点覆盖项目级总开关、工作区边界、审计与错误恢复。
  - 收口设置页、工作台与项目切换相关状态的同步细节，减少“前端显示状态”和“后端真实状态”不一致的问题。
  - 持续清理冗余代码、旧兼容链路和历史遗留实现，降低后续扩展成本。
- **进度状态**：基础稳定性已明显提升；当前重点转向文件写入能力落地后的权限、状态与体验收口。

## 2. 提示词优化与结构化输出
- **核心目标**：提升提示词稳定性、输出质量与工具调用一致性，减少非结构化输出带来的脆弱性。
- **具体方向**：
  - 基于当前共享 schema 与分层 Prompt，继续验证各 Agent 在真实项目问题中的角色化回答质量。
  - 探索更细粒度的 `reply` 协议设计，逐步从“纯文本回复”过渡到“更稳定的结构化前端承接”。
  - 继续打磨工具调用相关提示词，使工具使用、工作区上下文和执行反馈更自然地融入回复过程。
- **进度状态**：第一轮 Prompt 重写和 schema 基础层已完成；当前进入围绕真实使用场景的第二轮调优。

## 3. 前端的配合性优化
- **核心目标**：让前端更好承接多 Agent 工作流，提高可视化协作、状态感知与操作效率。
- **具体方向**：
  - 继续优化工作台与设置页的信息架构，提升项目栏、设置导航、工具展示和工作区管理的一致性。
  - 持续打磨前端外观与视觉表现，提升界面层级、节奏、留白、风格与整体统一性，让工作台更像成熟产品而不是功能面板拼接。
  - 完善工具执行过程与文件改动的前端可视化，使文件读写过程更透明、更可感知。
  - 在当前拆分后的前端结构上继续收敛 UI 层和逻辑层边界，降低页面级大组件的维护复杂度。
  - 逐步承接结构化结果、权限确认和文件上下文展示，而不只停留在聊天文本视图。
- **进度状态**：进行中；工作台、设置页和多模态输入基础已成形，当前重点转向工具过程展示和设置体验收口。

## 4. Unity 工程能力
- **核心目标**：从纯对话规划进一步走向“结合实际 Unity 工程文件的开发辅助”。
- **具体方向**：
  - 在现有工作区边界上继续完善 Unity 文件读写能力，优先做稳 `.cs` 等文本工程文件的创建、修改、补丁与删除流程。
  - 逐步补齐目录级操作、批量读取、工程阅读与检索、上下文文件集合与执行反馈，让 Agent 真正能围绕 Unity 工程工作，而不是只读单文件。
  - 推进 Unity 编辑器侧 MCP 接入，把 Unity Editor MCP Server / 工具层纳入近期能力建设，将放置 Prefab、移动物体、保存场景等操作逐步做成受控工具。
  - 逐步从“编辑器能力可调用”推进到“编辑器能力可组合”，为更复杂的场景操作、对象编排和自动化工作流打底。
  - 与工作区权限、执行确认和过程展示统一，避免编辑器工具成为绕过边界的特例通道。
  - 在保持安全边界的前提下，为后续 `.prefab`、`.unity` 等更高风险文件类型预留扩展位。
- **进度状态**：进行中；基础工作区边界与首批读写工具已落地，当前重点转向权限确认、过程展示、更多工程操作闭环，以及 Unity 编辑器侧能力的受控接入。

## 远期规划 (Long-term Vision)

> 以下为探索性方向，不设硬性排期，将根据近期计划推进情况逐步展开。

## 5. Agent 能力 Skills 化 (Skill-based Agent Architecture)
- **核心目标**：将 Agent 能力从大而全的 Prompt 中拆分出来，形成可组合、可扩展的能力模块。
- **具体方向**：
  - 设计统一的 Skill 接口与装配方式，降低能力扩展时对主流程的侵入。
  - 支持不同 Agent 在不同阶段按需加载 Skill，而不是依赖固定的一整套 Prompt。
  - 提升能力复用率，方便跨 Agent 共享最佳实践。

## 6. 更多引擎兼容与生态扩展 (Broader Engine Compatibility)
- **核心目标**：在 Unity 路线逐步稳定之后，把游戏开发能力扩展到更多引擎与生态工具，形成更统一的游戏开发工作台能力框架。
- **具体方向**：
  - 支持 Godot 等更开放的引擎方向，优先围绕文本化工程文件、项目阅读、检索、编辑器插件与 MCP 工具桥能力展开。
  - 将 UE 纳入长期兼容规划，优先考虑项目结构理解、配置与 C++ 工程辅助，再逐步评估编辑器侧与资源侧能力接入。
  - 与现有工作区、权限、工具目录、执行过程和设置页统一，避免每个引擎都长出一套彼此割裂的能力模型。

## 7. 游戏内模型能力接入 (In-Game LLM Integration)
- **核心目标**：让开发者可以通过工作台，低门槛地配置、测试并接入面向游戏运行时的大模型能力。
- **具体方向**：
  - 支持在工作台中定义游戏内可调用的 Agent / Prompt / Tool 组合，用于 NPC 对话、任务提示、剧情回顾、世界观问答等场景。
  - 提供适合游戏接入的配置与导出能力，例如运行时配置、调用方案、服务端接口约定和 Unity 侧接入骨架。
  - 逐步形成“工作台内配置与测试 -> 游戏工程接入 -> 运行时调试与迭代”的闭环，而不是只停留在开发期辅助。
  - 在设计上同步考虑成本控制、上下文边界、权限限制与运行时体验，避免把开发态能力直接粗暴搬进游戏运行时。

## 8. 文案与表达子能力 (Narrative & Copy Capability)
- **核心目标**：将文案生成作为策划 Agent 的子能力沉淀下来，让工作台可以围绕游戏工件、风格预设和用途约束，稳定地产出可直接使用的游戏文案。
- **具体方向**：
  - 支持基于 GDD、项目计划、角色设定、世界观说明等现有工件生成受约束的文案，而不是脱离上下文单独写作。
  - 支持风格 preset、用途分类、字数限制、禁用词、受众语气等配置，覆盖 UI 文案、系统提示、任务文本、NPC 对话初稿等不同场景。
  - 逐步形成“文案任务定义 -> 约束生成 -> 人工审阅 -> 可交付文案表”的工作流，优先沉淀结构化输入与结果导出能力。
  - 优先考虑 `YAML/JSON` 任务定义与 `CSV` 结果表的组合，为后续游戏工程接入、本地化和批量维护打基础。

## 9. 协作平台接入与可视化 (Collaboration Platform Integration)
- **核心目标**：让工作台可以连接 GitHub 等协作平台，并以更直观的方式呈现项目协作状态与研发节奏。
- **具体方向**：
  - 支持接入 GitHub 等常见代码协作平台，获取 issue、pull request、commit、review、CI 状态等协作信息。
  - 在工作台中以可视化方式展示项目协作概览，例如任务流转、提交活跃度、评审状态、分支变化与发布节奏。
  - 让 Agent 工作流与真实研发协作上下文建立连接，为后续任务分派、状态同步、项目复盘和团队协作辅助提供基础。

## 10. 外部 AIGC 能力集成 (External AIGC Integration)
- **核心目标**：让工作台可以作为游戏开发 AIGC 能力的统一入口，帮助用户更方便地访问外部生成式服务，而不是在多个网站之间来回切换。
- **具体方向**：
  - 支持集成外部图片、声音、3D 模型、原型设计等 AIGC 平台或服务，把常用入口与能力组织进工作台。
  - 为不同类型的外部 AIGC 能力提供更适合游戏开发的调用与跳转方式，例如素材生成、音频生成、模型资产获取、界面原型辅助等。
  - 逐步形成“在 Ludens-Flow 内选择能力 -> 跳转或调用外部 AIGC 服务 -> 回收结果继续进入项目工作流”的顺滑闭环。
  - 在设计上同步考虑账户体系、权限、结果回流、资产组织和后续可替换性，避免把工作台做成单纯的链接集合页。

## 11. Benchmark 与评测体系构建 (Benchmark & Evaluation)
- **核心目标**：建立适配 Ludens-Flow 的评测体系，用可重复、可比较的方式衡量工作流质量、稳定性与真实任务完成度。
- **具体方向**：
  - 构建项目自定义 workflow benchmark，覆盖需求澄清、定稿确认、工件生成、Review 回流、多模态输入与异常场景处理。
  - 参考真实任务型 benchmark 与工具调用 benchmark，评估多 Agent 协作、结构化输出和工具使用质量。
  - 沉淀统一的评估指标，例如阶段推进成功率、误跳转率、工件完整性、响应时延与演示稳定性。
