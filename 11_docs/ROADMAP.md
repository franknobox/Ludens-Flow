# Ludens-Flow 后续开发计划 (Roadmap)

## 1. 稳定优化
- **核心目标**：持续提升系统稳定性、状态流转一致性与整体可维护性，优先解决影响体验和可扩展性的基础问题。
- **具体方向**：
  - 优化多模态能力接入后的输入解析、错误兜底、数据流转与前后端协作体验。
  - 增加文件输入能力，优先支持纯文本、代码文件和常见文档输入，并完善解析、大小限制、编码处理与安全兜底。
  - 继续收口外置 Prompt、用户画像注入和结构化解析之间的边界，减少配置漂移和协议不一致问题。
  - 继续打磨多项目工作台的边界行为与管理能力，重点关注项目级上下文一致性、reset / migrate / active project 等稳定性细节。
  - 清理冗余代码与历史遗留实现，降低后续功能扩展成本,持续修复bug。
- **安装包与工程化收口（ludensflow）**：
  - 补齐 `agent_workbench/pyproject.toml`，明确构建后端、项目元数据、依赖与 `src/` 包发现规则。
  - 以 `ludensflow` 作为安装包名，保留 `ludens_flow` 作为 Python 导入包名，统一命名语义。
  - 提供标准入口：`ludensflow`（CLI）与 `ludensflow-api`（Web/API），逐步替代脚本式运行方式。
  - 将 `agent_workbench/api.py`、`agent_workbench/run_agents.py` 迁移为包内入口并移除 `sys.path` 注入；兼容期保留薄封装入口，避免中断现有使用。
  - 增加安装与启动 smoke test（如 `pip install -e ./agent_workbench` + 命令启动校验），保证本地、CI、部署环境行为一致。
- **进度状态**：进行中

## 2. 提示词 (Prompt) 优化与结构化输出 (Schemas)
- **核心目标**：提升提示词稳定性、输出质量与工具调用一致性，减少非结构化输出带来的脆弱性。
- **具体方向**：
  - 分阶段优化核心 Prompt，使不同 Agent 的职责边界、输出目标和约束条件更清晰。
  - 推动关键节点输出结构化，逐步引入统一的 Schema 约束。
  - 为后续工具调用、前端渲染和自动校验提供更稳定的数据接口。
  - 将 discuss 阶段的 `reply / state_updates / profile_updates / events` 收敛为共享 Schema 定义，减少各 Agent 重复维护协议文本。
  - 补齐统一的结构化解析、默认值归一和基础校验层，降低“看起来像 JSON 但无法稳定消费”的脆弱性。
  - 逐步统一 ReviewGate 等特殊结构，减少多套输出协议并存的情况。
  - 在外置 Prompt 基础上继续收敛模板组织方式，减少角色设定、画像注入和输出约束之间的重复拼接。
- **进度状态**：基础能力已接入，持续优化中

## 3. 前端的配合性优化
- **核心目标**：让前端更好承接多 Agent 工作流，提高可视化协作、状态感知与操作效率。
- **具体方向**：
  - 优化前端与后端状态同步机制，降低界面状态与实际工作流状态不一致的问题。
  - 继续改善 Agent 视图、文件视图、阶段提示、只读控制和输入反馈体验。
  - 在当前拆分后的前端结构上继续收敛 UI 层和逻辑层边界。
  - 支持文件上传与解析结果展示，优先覆盖纯文本、代码和文档类输入场景。
  - 增加前端定稿确认交互，例如确认按钮、状态提示和二次确认流程，降低误触发与误提交风险。
  - 为多模态输入、结构化输出展示与后续多项目工作台能力预留前端接口。
- **进度状态**：进行中

## 4. Unity 工程文件接入（基础）
- **核心目标**：从纯对话规划进一步走向“结合实际 Unity 工程文件的开发辅助”。
- **具体方向**：
  - 支持读取 Unity 项目目录结构与常见工程文件，增强上下文感知。
  - 优先覆盖 `.cs`、`.prefab`、`.unity` 等文件的分析、解释与辅助修改能力。
  - 逐步形成“读工程文件 / 改工程文件”的基础工作流，为后续更深的工程协作打底。
- **进度状态**：待启动

## 远期规划 (Long-term Vision)

> 以下为探索性方向，不设硬性排期，将根据近期计划推进情况逐步展开。

## 5. 工程阅读与检索能力 (Project Reading & Retrieval)
- **核心目标**：增强 Agent 对 Unity 工程与相关代码文件的阅读能力，支持更精准的定位、更快速的上下文收集与更稳定的分析辅助。
- **具体方向**：
  - 建立面向 Unity 项目的文件级检索能力，覆盖目录结构、代码文件与常见资源文件的快速定位。
  - 补齐符号级定位能力，例如类、方法、字段、命名空间、继承关系与调用线索的快速收集。
  - 逐步补齐 `.meta` GUID、`.prefab`、`.unity` 等 Unity 资源引用关系的解析与聚合能力。
  - 在此基础上引入 RAG 作为子项，用于语义召回、历史工件检索、Unity 文档与开发规范增强，而不是替代直接工程分析。
  - 为后续更高质量的 Coaching、Review、问题定位和方案比较提供依据。

## 6. Agent 能力 Skills 化 (Skill-based Agent Architecture)
- **核心目标**：将 Agent 能力从大而全的 Prompt 中拆分出来，形成可组合、可扩展的能力模块。
- **具体方向**：
  - 设计统一的 Skill 接口与装配方式，降低能力扩展时对主流程的侵入。
  - 支持不同 Agent 在不同阶段按需加载 Skill，而不是依赖固定的一整套 Prompt。
  - 提升能力复用率，方便跨 Agent 共享最佳实践。

## 7. Unity 编辑器 MCP 接入 (Unity Editor MCP Integration)
- **核心目标**：探索通过 Unity 编辑器侧工具能力，让 Agent 从“读/改文件辅助”进一步走向“受控编辑编辑器内容”。
- **具体方向**：
  - 探索通过 MCP 方式接入 Unity 编辑器侧能力，考虑实现一层 “Unity Editor MCP Server / 工具层”，把放置 Prefab、移动物体、保存场景等编辑器操作封装为可调用工具。
  - 在此基础上评估 Agent 通过 MCP 调用 Unity 编辑器工具的可行性，逐步从“读/改工程文件”扩展到“受控编辑场景与对象”。
  - 为后续更深入的项目协作和编辑器内自动化能力打基础。

## 8. Benchmark 与评测体系构建 (Benchmark & Evaluation)
- **核心目标**：建立适配 Ludens-Flow 的评测体系，用可重复、可比较的方式衡量工作流质量、稳定性与真实任务完成度。
- **具体方向**：
  - 构建项目自定义 workflow benchmark，覆盖需求澄清、定稿确认、工件生成、Review 回流、多模态输入与异常场景处理。
  - 参考真实任务型 benchmark 与工具调用 benchmark，评估多 Agent 协作、结构化输出和工具使用质量。
  - 沉淀统一的评估指标，例如阶段推进成功率、误跳转率、工件完整性、响应时延与演示稳定性。
