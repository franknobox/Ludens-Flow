# Ludens-Flow 游戏开发能力设计案

## 1. 文档定位

这份文档聚焦一个核心问题：

> Ludens-Flow 如何真正提升“做游戏”这件事的效率与质量，而不只是提供通用问答能力。

它是产品能力设计文档，不是底层实现细节文档。重点放在：

- 面向游戏开发的核心用户价值
- 当前阶段的能力边界
- 以 Unity 为优先落地方向的产品化路线
- 向 Godot、Blender 等后续能力扩展的统一思路

---

## 2. 核心目标

Ludens-Flow 的游戏开发能力，目标不是做一个“更会聊天的 AI”，而是把 Agent 放回真实工程系统中，让它围绕项目、工件、工具和运行边界工作。

要解决的核心问题包括：

1. 设计、策划、工程和评审之间经常脱节，导致反复返工。
2. 游戏项目的真实上下文难以进入对话系统，建议容易泛化。
3. 项目推进缺少可追踪、可回流、可复盘的稳定闭环。

对应的目标是：

- 让用户在统一工作台中完成讨论、定稿、开发、评审与回流。
- 让 Agent 不只理解 prompt，也理解项目、文件、工作区和工具边界。
- 让关键状态、关键操作和关键工件都能被追踪和延续。

---

## 3. 当前能力边界

### 3.1 已形成的能力

- 多阶段协作流程：Design / PM / Engineering / Review / Dev Coaching
- 多项目工作台与项目级状态管理
- 项目级工件体系：GDD、计划、评审报告、开发日志等
- 多模态对话输入：图片、文本、代码、文档附件
- 项目级工作区清单与安全边界
- 围绕工作区的首批文件读写能力

### 3.2 当前明确不做

- 不把系统定位成通用本地文件管理器
- 不在当前阶段开放高风险的无边界编辑器操作
- 不追求一次性覆盖所有引擎和 DCC 工具的全能力

### 3.3 当前阶段判断标准

只要系统能稳定支撑真实项目推进，就算能力成立。关键不在功能堆积，而在：

- 任务推进成功率
- 回流和评审质量
- 工件一致性与可追踪性
- 多轮迭代下的可持续性

---

## 4. 用户价值闭环

### 4.1 对独立开发者

- 把零散灵感和临时讨论，收敛成结构化推进过程
- 快速形成可执行方案，而不只停留在“想法很多”

### 4.2 对小团队

- 统一设计、计划、实现和评审之间的语言与产物格式
- 让关键决策和回流更可见，减少后期返工成本

### 4.3 对项目管理

- 从任务清单驱动，升级为状态驱动、工件驱动、流程可审计的推进方式
- 让关键节点（定稿、冻结、回流、恢复）具备清晰边界

---

## 5. 引擎与工具生态方向

### 5.1 Unity

Unity 是当前最优先落地方向。

重点不是做一堆零散小工具，而是形成一条产品闭环：

- 读工程文件
- 理解工程上下文
- 提供结合项目状态的建议
- 在安全边界内执行受控操作
- 逐步接入编辑器侧 MCP 能力

### 5.2 更多引擎兼容

在 Unity 路线逐步稳定之后，Ludens-Flow 应该逐步扩展到更多引擎与生态工具，而不是长期停留在单一引擎支持。

#### Godot

Godot 是后续非常值得推进的方向。

它的价值在于：

- 引擎与编辑器开源，可控性更高
- 核心项目文件多为文本格式，天然更适合 Agent 阅读、检索、diff 与 patch
- 编辑器插件与工具能力更容易和 MCP、受控工具链结合

Godot 方向重点可放在：

- `project.godot`
- `.tscn`
- `.gd`
- `.tres`
- Godot Editor Plugin / MCP 工具桥

#### UE

UE 值得纳入长期兼容规划，但优先级应低于 Unity 和 Godot。

它更适合先从“工程理解与协作辅助”切入，而不是一开始就做深度资源编辑。早期重点可以放在：

- `.uproject`
- `Config/*.ini`
- `Source/**/*.h`
- `Source/**/*.cpp`
- `*.Build.cs` / `*.Target.cs`
- `Plugins/`

后续再逐步评估编辑器侧工具桥、资源侧能力和更深的运行时协同方式。

### 5.3 Blender

Blender 更适合作为内容生产链的后续扩展方向。

它的价值不在于替代 Unity / Godot，而在于接入内容生产环节：

- 资产制作
- 场景搭建辅助
- 资源组织与导出流程

---

## 6. 产品化路线

### 6.1 第一阶段：可读上下文

- 能进入真实项目上下文
- 稳定读取关键文件与目录信息
- 让上下文真正影响讨论、规划和评审结果

### 6.2 第二阶段：可用建议

- 建议从泛化模板，升级为“结合当前项目结构的建议”
- 让设计、计划、实现和评审之间保持一致口径

### 6.3 第三阶段：可控改动

- 在可审计、可回退、可限制的前提下支持受控文件改动
- 保持“人做决策，系统做辅助”的关系

### 6.4 第四阶段：编辑器协同

- 在文件级能力和安全边界稳定之后，继续扩展到编辑器级协同
- 对 Unity 走 MCP / Editor Tool 路线
- 对 Godot 走 Editor Plugin / MCP 桥路线

---

## 7. 设计原则

1. 先让真实项目推进得动，再扩功能面。
2. 以协作效率为中心，不以工具数量为中心。
3. 安全边界必须清楚、可解释、可审计。
4. 每次迭代都要能回看、能恢复、能继续。
5. 先把一个引擎方向做稳，再扩到更多生态。

---

## 8. 前端体验要求

这些能力必须在界面上被用户“看见并理解”，而不是只存在于后端：

- 当前阶段和当前角色清楚可见
- 工件和工作区状态清楚可见
- 工具调用过程与文件改动过程清楚可见
- 关键状态变化和回流动作可追溯

一句话说：

> 游戏开发能力必须在工作台中表现为“可见、可控、可追踪”的过程体验。

---

## 9. 演进方向

### 9.1 近期

- 做稳项目级工作区和文件读写能力
- 做稳工具执行过程可视化
- 做稳 Unity 文件级与编辑器侧能力的受控接入

### 9.2 中期

- 从文件理解走向项目结构理解
- 提升评审与回流建议质量
- 让多 Agent 协作更依赖结构化计划和工程上下文，而不是单轮对话

### 9.3 远期

- 形成跨引擎的统一能力框架
- Unity、Godot、Blender 等能力逐步纳入同一工作台
- 让游戏开发工作台成为真正的“项目协作与能力接入中心”

---

## 10. 成功定义

当以下结果可以稳定出现时，可以认为这套“游戏开发能力”在产品层面成立：

1. 用户能在真实项目中持续推进，而不是只做演示型对话。
2. Agent 的建议明显受项目上下文影响，而不是泛化回答。
3. 文件、工件、工作区和流程状态保持一致并可追踪。
4. 系统可以在多轮协作中持续工作，而不是每轮都重新开始。

---

## 11. 总结

Ludens-Flow 的“游戏开发能力”本质不是某一个引擎插件或某一组工具，而是：

> 把游戏开发从碎片化协作，升级为可持续、可追踪、可迭代、可扩展的产品化工作流能力。

Unity 是当前第一落地点。  
Godot 是后续非常有潜力的开放引擎方向。  
UE 代表更高复杂度、更高价值的长期兼容方向。  
Blender 等工具则代表更广义的游戏生产链扩展。

最终目标不是支持更多名字，而是形成一套统一的游戏开发能力框架。
