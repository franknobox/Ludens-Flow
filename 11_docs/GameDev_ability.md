# Ludens-Flow 游戏开发能力设计稿

## 1. 文档定位

这份文档用于定义 Ludens-Flow 在 V3 阶段的**游戏开发能力边界**。

它不是某一个单独工具链的实现说明，而是一个更上位的能力设计稿，用来统一回答以下问题：

- 系统为什么要进入真实游戏项目环境
- 能力边界应该如何收拢
- 文件级能力、项目级能力、编辑器级能力分别是什么
- Unity 作为当前重点应如何落地
- Blender、DCC 工具、后续 MCP 接入如何纳入同一条演进路线

当前阶段以 Unity 为主要落地方向，但这份设计稿不再把能力限定为 “Unity 能力”，而是把 Unity 放在**游戏开发能力体系**中的第一优先实现对象。

---

## 2. 总体目标

Ludens-Flow 的目标不是给 Agent 一个“任意操作本地文件系统”的入口，而是让它在严格受控的前提下，逐步具备以下能力：

1. 进入真实游戏项目上下文
2. 阅读项目结构、脚本、资源关系与关键文件
3. 在受控范围内进行安全的文本级修改
4. 在未来进一步连接编辑器层能力
5. 最终形成一套面向游戏开发的、可扩展的能力系统

这套能力最终服务的不是“泛用编程助手”，而是：

- 游戏设计与工程协同
- 小团队 / 独立开发工作流
- Unity 项目辅助开发
- 后续可扩展到 Blender 等内容生产工具链

---

## 3. 能力分层

### 3.1 文件级能力

这一层面向“读懂工程文件”和“有限地修改文本文件”。

典型能力包括：

- 浏览目录
- 搜索文件
- 读取文本文件
- 读取文件片段
- 写入白名单文本文件
- 生成 patch / diff

这一层是当前最适合优先落地的部分，也是最容易做出稳定产品能力的一层。

### 3.2 项目级能力

这一层不只看文件本身，而是开始理解项目关系。

以 Unity 为例，典型能力包括：

- `.meta` / GUID 关系解析
- `.prefab` / `.unity` / `.asset` 中的资源引用关系
- MonoBehaviour 与资源挂载关系
- `asmdef` 边界
- 脚本、Prefab、Scene 之间的关联理解

以 Blender 为例，未来对应的能力可能包括：

- `.blend` 项目结构摘要
- 对象、集合、材质、贴图依赖关系
- 导出资源的组织关系
- 资产命名与目录约束检查

这一层的重点是“理解项目”，而不是直接控制编辑器。

### 3.3 编辑器级能力

这一层是更高风险、也更高价值的能力层。

它不再只是读写文件，而是通过编辑器 API、插件或 MCP 工具层进行受控操作。

Unity 方向的典型能力包括：

- 放置 Prefab
- 创建对象
- 移动对象
- 查询当前场景对象结构
- 保存场景

Blender 方向的典型能力包括：

- 创建对象
- 选择对象
- 应用材质
- 调整层级
- 导出指定资产

这一层应晚于文件级和项目级能力实现，不宜过早开放。

## 4. 设计原则

### 4.1 项目级沙箱优先

每个 Ludens 项目都应绑定自己明确的外部工程根目录，例如：

- `unity_root`
- `blender_root`
- 未来的 `game_root` / `content_root`

系统暴露给 Agent 的所有游戏开发工具，都必须限制在对应项目已绑定的根目录内。

这意味着：

- Agent 不应拥有全盘访问能力
- Agent 不应拥有任意路径跳转能力
- 外部工程访问边界由项目绑定配置决定

### 4.2 相对路径优先

对外暴露给 Agent 的路径语义应统一为：

- 以绑定根目录为根
- 所有目标文件和目录使用相对路径

例如：

- 合法：`Assets/Scripts/Player/PlayerController.cs`
- 非法：`C:\Project\Assets\Scripts\PlayerController.cs`

这样可以把路径校验集中在后端工具层，而不是把安全判断交给 Agent 或前端。

### 4.3 白名单工具优先

不应暴露“任意 shell”“任意文件系统操作”这类高风险入口。

应该暴露的是一组职责明确的白名单工具，例如：

- 列目录
- 找文件
- 读文件
- 写白名单文本文件
- 生成 patch
- 查询项目引用关系

### 4.4 先只读，后写入

能力开放顺序应当是：

1. 只读
2. 安全文本写入
3. 项目关系理解
4. 编辑器级控制

不要从一开始就把高风险写入或编辑器操作暴露给 Agent。

### 4.5 可审计、可确认、可回退

所有写操作都不应是黑箱。

至少需要具备：

- 写前能看到目标路径
- 写前可生成 diff / patch
- 写后返回修改摘要
- 关键操作有确认机制
- 写操作进入审计日志

## 5. 当前重点：Unity 能力

Unity 是当前最优先接入的外部游戏开发环境。

### 5.1 当前合理目标

在 V3 阶段，Unity 能力最合理的推进目标是：

- 让 Agent 能安全进入 Unity 项目目录
- 能浏览、搜索、读取脚本与工程文本文件
- 能逐步理解 Unity 项目中的资源与脚本关系
- 在后续阶段再扩展到安全写入与编辑器级能力

### 5.2 当前优先能力

优先落地：

- `unity_list_dir`
- `unity_find_files`
- `unity_read_file`
- `unity_find_text`
- `unity_read_files_batch`

在只读能力稳定后，再继续考虑：

- `unity_write_text_file`
- `unity_patch_text_file`
- `unity_create_text_file`

### 5.3 中期差异化能力

真正能拉开产品差异的，不只是“能读脚本”，而是：

- `unity_find_guid_references`
- `unity_get_prefab_scripts`
- `unity_get_scene_objects`

这些能力会让系统从“看文件”升级到“看项目结构”。

### 5.4 长期编辑器接入

更后续的方向是：

- Unity Editor MCP Server / 工具层
- 通过编辑器侧能力做受控操作

这应作为更晚阶段的演进方向，而不是文件级能力阶段的替代方案。

---

## 6. 未来扩展：Blender 与其他工具链

这份设计稿不应把能力固定死在 Unity 上。

未来如果 Ludens-Flow 继续向游戏内容生产延展，Blender 会是非常自然的下一站。

### 6.1 Blender 能力的预期位置

Blender 不应被视为“另一个散乱工具”，而应纳入同一套游戏开发能力框架：

- 文件级能力：浏览、读取相关配置与导出结果
- 项目级能力：理解对象、集合、材质、资产组织关系
- 编辑器级能力：通过插件 / MCP / 脚本接口进行受控操作

### 6.2 Blender 与 Unity 的关系

从产品视角，Unity 和 Blender 不是两套无关能力，而是：

- Unity 偏运行时工程与玩法实现
- Blender 偏内容生产与资产组织

两者未来都应服务于同一个目标：

> 让 Agent 能进入真实游戏开发环境，并在安全边界内帮助用户推进项目。

---

## 7. 工具接口建议

从系统实现上，推荐统一采用“受控工具层”的方式暴露能力，而不是直接给 Agent 任意文件系统权限。

工具可以按能力分组：

### 7.1 通用游戏开发工具

- `game_list_dir`
- `game_find_files`
- `game_read_file`
- `game_read_files_batch`
- `game_find_text`

### 7.2 Unity 专用工具

- `unity_find_guid_references`
- `unity_get_prefab_scripts`
- `unity_get_scene_objects`

### 7.3 Blender 专用工具

- `blender_list_assets`
- `blender_get_object_summary`
- `blender_get_material_links`

当前阶段不要求这些都立即实现，但应从接口设计上保留这类能力分层的空间。

---

## 8. 前端与交互建议

虽然本文重点是能力边界，但产品交互必须同步体现这些边界。

建议包括：

- 项目创建或项目设置中支持绑定外部工程目录
- 明确显示当前项目已绑定的开发环境
  - 例如 Unity Root / Blender Root
- 工具调用前后显示目标路径、操作类型和摘要
- 写操作前给出明确确认
- 审计信息可追踪

这样用户才能在产品层真正理解“系统现在能做什么，不能做什么”。

---

## 9. 推荐实现顺序

建议按以下顺序推进：

1. 固化项目级外部工程根目录绑定语义
2. 完善只读工具与路径安全边界
3. 补齐统一错误返回和审计日志
4. 增加检索类能力（全文搜索、批量读取）
5. 补充项目关系理解能力
6. 最后推进安全写入
7. 再考虑编辑器侧 MCP / 插件层接入

这个顺序的好处是：

- 每一步都能单独验收
- 能力边界始终清楚
- 不会过早暴露高风险操作

---

## 10. 收版判断标准

V3 阶段如果要认为“游戏开发能力”达到了可收版水平，建议至少满足：

1. 每个项目绑定的外部工程目录都有明确沙箱边界
2. 只读能力稳定可用
3. 路径输入统一为相对路径
4. 检索与批量读取能力至少具备雏形
5. 工具调用结果有统一反馈和错误提示
6. 写入能力如开放，必须具备白名单、确认与审计机制
7. Unity 方向的能力已经形成完整的第一条产品闭环

---

## 11. 总结

这份设计稿的核心判断是：

Ludens-Flow 后续不应只做“Unity 能力”，而应逐步形成一套面向游戏开发全过程的能力系统。

这套系统的合理演进路线应当是：

- 先做项目级沙箱
- 再做只读与检索
- 再做项目关系理解
- 再做安全写入
- 最后接入编辑器级能力

Unity 是当前最优先落地的方向，
Blender 是未来很自然的扩展方向，
两者都应被纳入同一套“游戏开发能力”框架中来思考和实现。
