# Ludens-Flow Unity 能力设计稿

## 1. 文档定位

这份文档用于定义 Ludens-Flow 在 V3 阶段的 Unity 能力设计边界。

它的目标不是直接规定所有实现细节，而是先把以下几件事定清楚：
- Unity 能力到底服务什么场景
- 安全边界应该收在哪
- 工具层能力应如何分层
- 文件级能力与后续 MCP / 编辑器能力是什么关系

后续相关实现、测试和前端交互，都应尽量以本文为对照基线推进。

## 2. 设计目标

Unity 能力接入的核心目标，不是让 Agent 获得一个“任意文件系统操作入口”，而是让它在严格受控的前提下，具备进入实际游戏项目、读取上下文、进行有限文件操作，并为后续编辑器侧能力接入预留扩展基础。

这一层希望解决的问题包括：
- Agent 只停留在文档和工件层，无法真正进入游戏项目现场
- 缺少对 Unity 项目文件的稳定读取与定位能力
- 缺少受控的、可追溯的文件操作边界
- 缺少从“工程文件辅助”走向“编辑器工具能力”的演进路径

## 3. 核心原则

### 3.1 项目级沙箱优先

每个 Ludens 项目只能绑定一个明确的 `unity_root`，并将它作为该项目所有 Unity 工具的唯一根目录。

所有 Unity 相关工具，无论是读取、搜索还是后续写入，都只能在这个根目录内部工作。

这意味着：
- Agent 不拥有全盘访问能力
- Agent 不拥有任意目录跳转能力
- Unity 工具的能力边界，由项目绑定的 `unity_root` 决定

### 3.2 相对路径优先

Unity 工具层不应接受绝对路径作为正常输入。

对外暴露给 Agent 的路径语义应统一为：
- 以 `unity_root` 为根目录
- 所有目标文件和目录只通过相对路径描述

例如：
- 合法：`Assets/Scripts/Player/PlayerController.cs`
- 非法：`C:\\GameProject\\Assets\\Scripts\\PlayerController.cs`

这样做的目的，是让所有边界检查都集中在后端，而不是把路径安全交给 Agent 或前端判断。

### 3.3 白名单工具优先

Unity 能力不应以“任意执行命令”或“任意文件系统操作”的形式暴露给 Agent。

应当采用白名单工具形式，将能力拆成一组职责明确的工具，例如：
- 查目录
- 找文件
- 读文本文件
- 写文本文件
- 生成 patch
- 读取 Unity 资源引用信息

不要提供这类高风险入口：
- 任意 shell 执行
- 任意绝对路径写入
- 无约束批量删除
- 无约束二进制改写

### 3.4 分级写权限

Unity 能力不应从第一天起就开放完全写权限。

更合理的阶段划分是：
- `read-only`
- `safe-write`
- `editor-mcp`

其中：
- `read-only` 用于目录浏览、文件搜索、文本读取
- `safe-write` 仅允许受控修改部分文本类文件
- `editor-mcp` 用于后续更高层的编辑器侧操作，如场景、Prefab、对象编辑

### 3.5 可审计与可回退

任何写操作都不应是黑盒。

至少应满足：
- 写前可生成 diff 或 patch
- 写后可返回修改摘要
- 关键操作写入审计日志
- 高风险操作在前端有明确确认

## 4. 安全边界设计

## 4.1 `unity_root` 绑定规则

建议将 `unity_root` 绑定视为项目创建阶段的重要输入之一。

推荐语义如下：
- 创建项目时可填写 Unity 工程目录
- 成功绑定后，作为该项目唯一可访问的 Unity 根目录
- 普通运行流程中默认只使用这个目录
- 如果需要修改绑定，应通过明确的“重新绑定”操作进行

这样做的优势是：
- 项目上下文更稳定
- Agent 不会在运行中随意切换工程
- 安全边界从一开始就明确

## 4.2 路径解析规则

后端在处理任意 Unity 工具请求时，应统一遵循：

1. 先取出当前项目绑定的 `unity_root`
2. 将输入路径视为相对路径
3. 与 `unity_root` 拼接后再 `resolve()`
4. 确认最终路径仍然位于 `unity_root` 内部
5. 若越界则拒绝

额外建议：
- 禁止 `..` 越界路径
- 拒绝直接传入绝对路径
- 对符号链接、junction、reparse point 保持保守策略

## 4.3 文件类型白名单

在 V3 阶段，建议将“可写”范围限制在文本类文件，并且以白名单方式控制。

建议优先支持：
- `.cs`
- `.json`
- `.txt`
- `.md`
- `.yaml`
- `.yml`
- `.asmdef`

建议先不要开放直接写入：
- `.meta`
- `.prefab`
- `.unity`
- 二进制资源
- 图片、音频、模型等内容资源

原因是这些文件虽然很多也是文本或 YAML，但它们和 Unity 的资源系统、GUID 关系、序列化结构耦合更深，写错成本更高。

## 4.4 操作级边界

建议把工具操作分成三档：

### A. 低风险

- 列目录
- 搜索文件
- 读取文本文件
- 读取部分上下文片段

这些能力应优先落地，并尽快做稳定。

### B. 中风险

- 新建文本文件
- 覆盖文本文件
- 生成 patch
- 应用有限 patch

这些能力适合在文件边界和审计机制明确后再开放。

### C. 高风险

- 删除文件
- 移动文件
- 批量改写
- 修改 `.unity/.prefab/.meta`
- 直接作用于场景和对象

这些能力不建议在早期文件工具阶段开放，而应延后到更明确的安全与确认机制成熟后处理。

## 5. 能力分层设计

### 5.1 第一层：工程文件只读能力

这一层是最优先落地的部分。

目标包括：
- 浏览目录结构
- 搜索目标文件
- 读取代码与配置内容
- 为 Agent 提供稳定的项目上下文

典型工具可以包括：
- `unity_list_dir`
- `unity_find_files`
- `unity_read_file`
- `unity_read_snippet`

这一层的重点不是“改文件”，而是“先把项目读懂”。

### 5.2 第二层：受控文本写入能力

这一层用于支撑更实际的开发辅助，但仍然保持保守边界。

建议工具包括：
- `unity_write_text_file`
- `unity_patch_text_file`
- `unity_create_text_file`

配套规则：
- 只允许白名单文本文件类型
- 路径必须在 `unity_root` 内
- 返回 diff / patch 摘要
- 高风险覆盖需要确认

### 5.3 第三层：工程关系理解能力

当基础读写能力稳定后，应继续补强 Unity 项目的上下文理解能力，例如：
- `.meta` GUID 对应关系
- `.prefab / .unity` 中的资源引用关系
- `MonoBehaviour` 脚本与资源挂载关系
- `asmdef` 边界

这一层更偏“读懂工程关系”，而不是直接“编辑工程对象”。

### 5.4 第四层：Unity 编辑器 MCP 能力

更后续的方向，应通过 MCP 将能力从“读/改工程文件”延伸到“受控调用编辑器工具”。

典型能力包括：
- 放置 Prefab
- 移动物体
- 创建对象
- 保存场景
- 查询当前场景对象结构

这一层不应与文件读写层混在一起，而应作为更高层、更高风险、更强交互约束的一组工具来设计。

## 6. 对 Agent 的暴露方式

对 Agent 来说，Unity 能力不应表现为“你可以随意访问工程目录”，而应表现为“你可以调用一组受控工具”。

建议以能力声明的方式告诉 Agent：
- 当前是否绑定了 Unity 工程
- 当前可用的是只读工具还是读写工具
- 当前不允许哪些操作

这样可以减少 Agent 因为能力理解不清而胡乱尝试。

## 7. 前端与交互建议

虽然本文重点是能力边界，但前端也需要配合这些安全规则。

建议包括：
- 项目创建时支持输入 Unity 工程目录
- 当前项目页明确显示 Unity 绑定状态
- 工具调用前，用户能看到目标路径和操作类型
- 写操作、高风险操作、覆盖操作应有明确确认
- 工具调用结果应展示“操作了哪个相对路径、成功与否、返回了什么摘要”

这可以让用户在产品层真正感受到系统边界，而不是把安全规则隐藏在后端内部。

## 8. 推荐实现顺序

如果按稳妥路线推进，建议顺序如下：

1. 固化 `unity_root` 项目级绑定语义
2. 完善只读工具与路径边界校验
3. 补齐统一错误返回和审计日志
4. 开放受控文本写入工具
5. 增加工程关系解析能力
6. 最后再推进 Unity 编辑器 MCP 工具层

这样做的好处是：
- 每一步都能独立验证
- 不会一开始就把高风险能力暴露给 Agent
- 安全边界和产品体验可以同步演进

## 9. 收版判断标准

Unity 能力在 V3 阶段的“可收版”标准，建议至少包括：

1. 每个项目的 Unity 能力都严格收束在项目级 `unity_root` 内。
2. Unity 工具只接受相对路径，不接受任意绝对路径。
3. 只读工具已经稳定可用，并具备统一错误返回。
4. 受控写入能力如开放，则必须具备白名单、审计与确认机制。
5. 文件能力与后续 MCP 编辑器能力的边界已经明确。

## 10. 总结

Unity 能力的关键，不在于“尽快给 Agent 更多操作权”，而在于先把边界和层次做对。

对 Ludens-Flow 来说，更合理的路径是：
- 先做项目级沙箱
- 再做受控只读
- 再做受控写入
- 最后才走向编辑器侧 MCP 能力

只有这样，Unity 能力才会真正成为一个可以放心扩展、可以长期维护、可以和产品体验共同演进的系统层能力，而不是一个短期可用但边界模糊的危险入口。
