模块一：用户洞察与问题定义

目标用户

对游戏制作感兴趣的大学生、独立开发者及希望利用 AI 辅助创作的年轻创作者群体。他们通常具备一定的编程或设计基础，但缺乏系统化游戏开发经验，渴望以更低门槛进入游戏创作领域。

用户痛点

  1. 开发门槛高：游戏开发涉及策划、美术、程序、测试等多学科交叉，大部分高校未开设系统性的游戏开发专业，自学路径零散，入门成本极高。
  2. 工具链复杂：传统游戏引擎（Unity、Unreal、Godot）及配套工具种类繁多，学习周期长、软件授权成本高，初学者容易在配置环境阶段就放弃。
  3. AI 工具碎片化：现有 AI 工具（对话助手、代码生成、图像生成）各自为政，缺乏与游戏开发流程的深度整合，难以形成从需求到实机的闭环工作流。

使用场景

  • 初学游戏开发：通过结构化工作流和 Agent 辅导，零基础用户也能按步骤完成首个游戏原型。
  • Game Jam：48 小时极限开发场景下，AI 辅助快速产出 GDD、排期表、技术方案，压缩前期筹备时间。
  • 快速原型验证：创意阶段快速生成可运行原型，验证核心玩法后再决定是否投入正式开发。
  • 长期项目辅助：作为团队的"AI 副手"，持续参与项目迭代、代码审查、文档维护。

模块二：产品方案设计

产品概述

Ludens-Flow

Ludens 取自 Johan Huizinga《Homo Ludens》（游戏的人），意为"游戏者"——游戏是人的天性，也是创作的本源。
Flow 既指工作流的结构化流转（Design → PM → Engineering → Review），也指向创作者进入心流（Flow State）的理想体验。
合起来，Ludens-Flow 是让 AI 与创作者共同进入"游戏化开发心流"的结构化工作台。

Ludens-Flow 是一个面向游戏开发的 AI 多智能体结构化工作台。它将"自由聊天的 AI"转化为"有角色、有分工、有产出、有边界"的工程协作者：通过 Design、PM、Engineering、Review 等多个专业 Agent 的协作流水线，驱动用户从需求讨论、策划定稿、工程实施到评审优化的完整开发流程；同时通过 MCP 协议直接连接 Blender、Unity 等外部引擎，让 AI 不仅能"动嘴"写文档，更能"动手"操作实机。

核心功能

功能说明
角色化 Agent 辅导Design Agent 讨论玩法、PM Agent 排期收敛、Engineering Agent 输出技术方案、Review Agent 交叉审查，每个 Agent 有明确职责边界和专业提示词。
模拟开发全流程内置 GDD_DISCUSS → PM_DISCUSS → ENG_DISCUSS → REVIEW → DEV_COACHING 标准流水线，支持阶段回流与人工决策节点。
工件读写与管理自动生成并维护 GDD.md、PROJECT_PLAN.md、IMPLEMENTATION_PLAN.md、REVIEW_REPORT.md 等结构化工件，支持人工编辑与版本冻结。
MCP 引擎集成通过 Model Context Protocol 连接 Blender（已实机验证）、Unity、Godot、Unreal，支持场景操作、对象创建、文件保存等实机能力。
Skills 扩展系统支持外部导入自定义 Skill（skill.json + prompt.md），项目级启用，让 Agent 能力可扩展、可沉淀。
GitHub 集成前端可视化 GitHub 协作入口，连接代码仓库与开发流程。
AIGC 快捷入口统一接入图像生成、文字创作等外部 AIGC 服务，避免工具跳转。
LLM 快速接入支持 OpenAI、Moonshot、DeepSeek、OpenRouter 等多厂商模型，项目级路由配置，不同 Agent 可分配不同模型。
文案生成与导出内置文案加工台，支持外部参考文件、实时生成状态展示、Markdown / CSV 导出。
Coding 辅助支持代码文件读写、目录创建、Patch 应用、路径逃逸拦截等受控工程操作。

核心差异一句话：我们不是让 AI 替你做完游戏，而是让 AI 成为你的结构化开发伙伴——有分工、有产出、有边界、能实机操作，你在每个环节都深度参与并从中学习。

模块三：AI 原生能力说明

AI 核心能力

Ludens-Flow 构建了一套面向复杂创意工程的认知架构级 AI 系统，其核心能力不仅限于大语言模型的文本生成，而是将多模态理解、工具使用（Tool Use）、多智能体协作（Multi-Agent Collaboration）与外部系统操控（MCP 协议）进行深度融合：

  1. 认知角色化（Cognitive Role Specialization）：系统内嵌设计、项目管理、工程实施、质量审查四类认知角色，每个 Agent 搭载领域专属提示架构与知识边界，实现从"通用对话"到"专业协作"的跃迁。
  2. 结构化推理与工件生成（Structured Reasoning & Artifact Synthesis）：Agent 的输出不再是自由文本，而是严格遵循 GDD、PROJECT_PLAN、IMPLEMENTATION_PLAN 等标准化工件 Schema 的结构化内容，确保可追踪、可审阅、可执行。
  3. 具身化工具操控（Embodied Tool Manipulation via MCP）：通过 Model Context Protocol，AI 获得对外部数字环境的"手脚"——不仅能读取场景状态，更能创建对象、保存文件、修改工程，实现从"数字建议"到"物理操作"的闭环。
  4. 动态上下文工程（Dynamic Context Engineering）：系统实时聚合用户画像、项目历史、Skill 注入、模型路由配置等多维上下文，为每个 Agent 决策提供精准的情境感知，而非基于孤立提示的浅层响应。
  5. 协作式质量控制（Collaborative Quality Control）：Review Agent 对全流程工件进行交叉审查，结合人工决策节点（POST_REVIEW_DECISION），形成"AI 自检 + 人工把关"的双重质量闸门。

AI 如何解决痛点

用户痛点传统方案局限Ludens-Flow AI 解法
开发门槛高，缺乏系统学习路径看教程、读文档，知识点零散多 Agent 流水线即学习路径：用户在参与 Design → PM → Engineering 的过程中，自然理解游戏开发的标准流程与交付物要求，边做边学。
工具链复杂，学习周期长需要分别掌握引擎、IDE、版本控制、AI 工具统一工作台 + MCP 集成：AI 代理操作复杂工具链，用户通过自然语言与结构化界面与系统交互，无需精通每个软件的底层操作。
AI 工具散乱，场景适配不够对话助手不懂游戏开发流程，编程助手不懂策划需求流程原生集成：AI 能力内嵌于游戏开发工作流的每个阶段，不是外挂工具，而是 workflow 的一部分。GDD 生成后直接流入 PM 排期，技术方案定稿后可直接触发引擎操作。

去掉 AI，这个产品还能成立吗？

不能。Ludens-Flow 的本质价值在于AI 作为结构化协作节点的重新定位——没有多 Agent 的认知分工，就没有流水化的专业辅导；没有 LLM 的上下文理解，就没有从自然语言需求到结构化工件的转化；没有 MCP 的工具操控能力，就没有从文档到实机的闭环。AI 不是锦上添花的功能模块，而是产品的骨架与神经系统。

层级技术选型作用

模块四：落地可行性

落地可行性

  • 技术实现路径清晰：核心架构（Graph 编排、Agent 基类、MCP 适配器、Tool Registry）已稳定运行，Blender MCP 已通过实机验证。
  • 开发节奏可控：当前处于 v3 preview 阶段，后端测试覆盖 86 项（全通过），前端构建通过，具备持续迭代基础。
  • 所需资源合理：开源项目形态，依赖社区 LLM API 与本地引擎，无需重资产投入。

商业化思考

模式一：开源社区 + 赞助生态

保持核心项目开源，通过 GitHub Sponsors、企业定制服务、技术咨询获得持续资金支持。以开发者社区口碑建立行业影响力，形成"开源产品 → 社区反馈 → 能力增强"的正循环。

模式二：融入腾讯 PCG 内容矩阵

作为 PCG 体系内新型 AI 创作基础设施：

  • 定位：面向游戏创作者（从学生到独立团队）的 AI 原生开发工作台
  • 场景：与 PCG 现有内容生产管线对接，为内部项目或外部创作者提供 AI 辅助原型开发、文案生成、多引擎协作能力
  • 优势：填补 PCG 在"结构化 AI 开发工作流"领域的空白，区别于单一对话或代码生成产品，形成差异化 AI 产品线
