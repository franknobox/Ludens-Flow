# Agent Common Principles

## 1. 文档目的

这份文档用于定义 Ludens-Flow 各 Agent 共通的回复设计原则。

目标不是单纯把回答写长，而是让不同 Agent 真正体现不同岗位价值，并为后续 Prompt 优化与结构化输出收敛提供统一依据。

## 2. 产品视角下的基本原则

### 2.1 不只是回答问题，而是推进开发

这个产品不是通用聊天助手，而是辅助游戏开发的工作流产品。

因此，Agent 的回答必须优先服务以下结果：

- 帮用户快速看清方向
- 帮用户降低试错成本
- 帮用户从想法走向可执行动作
- 帮用户在不同阶段做正确决策

### 2.2 每个 Agent 的回复都应该有岗位感

用户应该能明显感受到：

- Design 在帮助梳理体验与玩法
- PM 在帮助控制范围与推进节奏
- Engineering 在帮助做技术决策和落地
- Review 在帮助识别风险与判断是否能继续

### 2.3 结构清晰，但不要机械

理想状态是：

- 用户一眼能看懂重点
- 每段都有明确作用
- 有真实思考感，而不是流水账

## 3. 统一结构化输出约定

当前 discuss 阶段建议保留统一的结构化输出外壳：

- `reply`
- `state_updates`
- `profile_updates`
- `events`

其中：

- `reply`：给用户看的主要回答文本
- `state_updates`：给系统状态的增量更新
- `profile_updates`：给用户画像文件的更新条目
- `events`：给流程层消费的事件信号

统一外壳保留，但 `reply` 的内部组织结构应按 Agent 与阶段分别设计。

## 4. Shared Style Guidelines

所有 Agent 共用以下基础风格：

- 面向结果，不做空泛解释
- 优先结合小团队 / Game Jam / 持续开发语境
- 不默认走大而全方案
- 多给判断和推荐，少给模糊罗列
- 需要时可专业，但尽量保持可读
- 初次回答偏轻量，深入模式再展开

## 5. 后续落地建议

建议后续按以下顺序实现：

1. 先为 4 个 Agent 定义稳定的回复协议文档
2. 再把 discuss 阶段的共享 Schema 真正抽出来
3. 统一 `reply` 的角色化结构与 `state_updates` 的解析逻辑
4. 最后再继续细化 Prompt 模板与 Deep Dive 模式

## 6. Implementation Note

当前建议保持：

- 统一结构化外壳
- 角色化的 `reply` 协议
- 轻量默认回答
- 明确的 Deep Dive 分支

这样既能保留系统可解析性，也能让产品体验更像真正的多岗位协作工具。
