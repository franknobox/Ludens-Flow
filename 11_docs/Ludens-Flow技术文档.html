1. 项目定位

Ludens-Flow 是一个面向游戏开发流程的多 Agent 工作台。它不是单次对话式的“游戏点子生成器”，也不是只围绕代码仓库工作的 Coding Agent，而是把 Agent 放回真实游戏开发流程里：围绕项目、工件、状态、工具、权限和人类决策组织协作。

项目希望解决的问题是：当一个新人或独立开发者想用 AI 做游戏时，AI 不应该只给一堆建议，而应该帮助用户形成可维护的开发资产，并能在后续持续参与项目。

Innovation

多 Agent + 状态机

Design、PM、Engineering、Review 四类 Agent 通过状态机协作，不把所有职责压给一个通用助手。

Robustness

工件与权限边界

输出落到 Markdown/JSON 工件；文件写入、MCP 操作、Skills 都进入项目级权限边界。

Human Interaction

人在环路中决策

Agent 负责流程、指导、执行，人负责创意、取舍、Review 回流和最终确认。

1.1 核心创新点

第一，项目不是把多个 Agent 简单串起来，而是用状态机组织游戏开发流程。
core/router.py 维护阶段与显式 action，core/graph.py 根据阶段调度 Design、PM、Engineering、Review 节点。用户不再依赖“输入任意内容继续”，而是通过定稿、回流、进入开发等明确动作推动流程。

第二，输出不只停留在聊天记录，而是沉淀为可维护的项目工件。
capabilities/artifacts/artifacts.py 负责 GDD、PROJECT_PLAN、IMPLEMENTATION_PLAN、REVIEW_REPORT 等工件写入，state_models.py 记录 owner、version、hash 和 updated_at。对话、工件、项目状态被拆开管理，便于回看、修改和继续开发。

第三，把游戏引擎、创作工具和项目资料抽象成面向游戏制作的统一能力层。
Unity、Godot、Blender、Unreal 等外部 MCP 的原始工具并不直接暴露给 Agent，而是通过 capabilities/mcp/adapter.py 映射为 engine_* 这类游戏开发语义能力；文案加工、AIGC 入口、GitHub 协作视图和 Skills 则作为不同 Agent 的子能力进入工作台。创新点在于把“会聊天的模型”变成“理解项目、能读工件、能接引擎、能复用流程”的游戏开发协作系统。

1.2 与其他游戏开发辅助方案的比较

方案优势用于游戏开发时的缺口Ludens-Flow 的定位
Gemini 等通用对话助手知识覆盖广，讨论成本低缺少项目状态、阶段边界、工件版本和工具权限将对话放入项目工作流，持续维护工件和状态
Claude Code / Codex 类 Coding Agent工程执行、文件读写、diff、命令反馈强主要面向代码仓库，不天然覆盖 GDD、文案、关卡、引擎对象和新人学习吸收工程 harness 思路，但面向游戏开发全链路扩展
Gambo 等一键生成型产品快速出原型过程较黑盒，用户控制与学习空间有限强调可解释过程、人类决策和可回流工件
TapTap 制造等创作平台入口友好，生态集中更偏平台内创作，工程/工具链自由度受限本地优先，可接入用户自己的引擎、仓库和文件系统
传统引擎软件组合能力完整，可做正式项目学习路径陡峭，新人不知道下一步做什么用 Agent 工作流、上下文管理和工具调用降低流程门槛

2. 设计意图

当前宣传的“AI 做游戏”：很多产品展示的是一键生成一个 Web 小程序 demo，但这并不等于真正参与游戏制作。对游戏开发感兴趣的入门者，很难在这种黑盒流程里理解玩法为什么这样设计、工程为什么这样拆、资源为什么这样组织；同时，现阶段生成式开发能力本身也有限，直接把完整游戏交给模型一次性生成并不稳定。

核心是让 Agent 参与游戏内容生成、关卡设计、NPC 行为逻辑和开发流程。Ludens-Flow 的设计判断是：游戏制作 Agent 的价值不应只体现在某一次生成，而应体现在一个可持续推进的开发系统里。

Practice

来自游戏开发实践的判断

小型游戏开发是一组连续决策：想法是否能形成 MVP、玩法先做哪部分、工程结构怎么搭、什么时候该回流修正、文案和代码如何进入真实项目。通用聊天 AI 能回答部分问题，但缺少稳定的项目记忆和执行边界。

Human-in-the-loop

人在环路中的分工

Agent 负责流程、指导、执行和候选方案；用户负责思考、想法、取舍和最终决策。这样既能降低入门门槛，又不会把创作主体交给黑盒。

2.1 工具整合思路

真实游戏开发需要同时使用游戏引擎、代码编辑器、建模工具、GitHub、AIGC 网站和文档系统。Ludens-Flow 的工作台层将这些能力整合为项目级入口，后端能力层再通过统一权限与 Adapter 把它们暴露给 Agent。

3.2 状态机内核

当前流程通过 core/router.py 和 core/graph.py 共同实现。Router 只负责判断下一阶段，Graph 负责执行对应 Agent、合并状态、写工件和自动推进非交互节点。

图 2：核心状态机

3.2.1 关键实现方法

状态机内核的重点不是枚举阶段名称，而是把“当前阶段”“用户显式动作”“Agent 执行结果”和“项目状态落盘”拆成稳定接口。这样 Agent 不需要在上下文里记住流程，前端也不需要靠用户输入任意文本来推进阶段。

显式动作驱动阶段跳转

agent_workbench/src/ludens_flow/core/router.py

前端按钮发送 explicit_action，Router 根据当前 phase 查询允许动作。未声明的动作直接留在当前阶段，避免自由文本误触发流程推进。

allowed = transition_table[current_phase]
if explicit_action not in allowed:
    return stay(current_phase)

next_phase = allowed[explicit_action]
return route_to(next_phase)

状态持久化只保存流程事实

agent_workbench/src/ludens_flow/core/state/state_models.py

LudensState 不把工件正文塞进状态文件，而是保存阶段、ReviewGate、短期对话、完整流水和工件元数据。工件正文由 artifacts 层管理，状态文件负责恢复流程和判断下一步。

state = {
  phase,
  review_gate,
  artifact_frozen,
  chat_history,
  transcript_history,
  artifact_meta: { owner, version, hash, updated_at }
}

Graph 合并 Agent 结果并推动副作用

agent_workbench/src/ludens_flow/core/graph.py

Router 只决定路由，Graph 负责真正执行：组装上下文、调用 Agent、合并结构化结果、写入工件、发送 SSE 事件，并在非交互节点中继续推进。

result = agent.run(context)
state.merge(result.state_updates)
artifacts.commit(result.artifact_updates)
events.emit(result.events)

  • 纯文本路径支持分段流式输出，不是一字一字刷新。
  • 结构化解析统一走 extract_structured_json_object()。

3.5 后端能力层

后端能力层位于 agent_workbench/src/ludens_flow/capabilities。它的目标是把“Agent 可以做什么”收敛成可审计、可组合、可限制的系统能力，而不是让每个 Agent 自己随意处理文件、引擎或外部平台。

artifacts

管理 GDD、PROJECT_PLAN、IMPLEMENTATION_PLAN、REVIEW_REPORT、DEVLOG、NOTES 等项目工件，记录 actor、reason、version、hash 和日志。

ingest

将图片、文本、代码、PDF 等输入转成模型可消费上下文，并向前端返回 warning，避免附件静默失败。

workspaces

统一路径审核，解析项目工作区、检查可写开关、阻止路径逃逸；所有文件写入必须经过项目级边界。

tools

注册 web_search、workspace_*、engine_*、skill_* 等 Agent 工具，并标记权限和类别。

mcp

连接 Unity、Godot、Blender、Unreal MCP，并映射到统一 engine_* 能力，不直接暴露原始 MCP 工具。

copywriting

Design Agent 子能力，读取 GDD 与外部资料生成游戏文案，输出候选文本、Markdown 和 CSV。

skills

导入外部 Skill，按项目启用，并支持自我沉淀；运行时只注入当前项目启用且适用的 Skill。

github

只读接入 GitHub 协作信息，辅助 PM Agent 理解 issue、PR、commit、CI 等研发节奏。

capabilities/tools/registry.py 是 Agent 工具调用入口。工具目录会标记 category、是否需要 workspace、是否会写文件；写入类操作统一发出 permission_required、file_changed、tool_completed 等事件，前端再展示确认和执行过程。

MCP Adapter 不直接把外部 MCP 工具列表暴露给 Agent，而是先映射成稳定的 engine_* 能力。这样 Unity、Godot、Blender 的底层工具名不同，但上层 Agent 学到的是同一套能力语义。

4. 功能中心

功能中心对应前端 agent_workbench/web/src/features。它更接近用户实际看到和使用的产品层：项目工作台、AIGC 快捷入口、文案加工、引擎 MCP、Skills、GitHub 协作视图、游戏内模型接入和设置管理。

4.1 Agent 设计：人设、职责边界与子能力

Ludens-Flow 没有把所有工作压给一个“全能助手”，而是把游戏开发流程拆成四个稳定角色。每个 Agent 都有明确的人设语气、阶段职责、工件 owner 和可使用的子能力；Graph 层再根据 Phase 决定当前应该调用谁。

Dam / Design

策划伙伴，负责把模糊创意收束为可制作的玩法方案。主要工作在 GDD_DISCUSS 与 GDD_COMMIT，产出 GDD.md、游戏标签和文案候选。

职责边界是核心体验、题材、系统设定和 MVP 边界；不负责排期和工程落地细节。子能力包括文案加工台：读取 GDD、外部参考和用户约束，生成台词、物品描述、世界观片段等可导出的游戏文案。

capabilities/copywriting, core/schemas/copywriting.py

Pax / PM

项目伙伴，负责范围控制、里程碑和开发节奏。主要工作在 PM_DISCUSS 与 PM_COMMIT，产出 PROJECT_PLAN.md、ChangeRequest 和协作概览。

职责边界是决定什么先做、什么砍掉、如何拆阶段；不替 Engineering 输出具体代码方案。子能力包括 GitHub 协作视图：只读获取 issue、PR、commit、CI 等协作信息，辅助判断研发节奏。

capabilities/github

Eon / Engineering

工程合伙人，负责工程结构、实现路径和持续开发指导。主要工作在 ENG_DISCUSS、ENG_COMMIT 与 DEV_COACHING，产出 IMPLEMENTATION_PLAN.md、DEVLOG.md 和工具执行结果。

讨论阶段只确定架构和路线；进入持续开发后才执行文件写入、Patch、引擎 MCP 等操作。子能力包括工作区文件工具、Unity / Godot / Blender / Unreal MCP 统一能力和工程类 Skills。

capabilities/workspaces, capabilities/mcp, capabilities/tools

Revs / Review

评审关卡，负责从设计、工程和范围健康度审查项目。主要工作在 REVIEW 与 POST_REVIEW，产出 REVIEW_REPORT.md、ReviewGate JSON 和回流目标。

职责边界是不继续扩写方案，而是判断 PASS / REQUEST_CHANGES / BLOCK，并指出应回流到 GDD、PM 或 ENG。Review 结果驱动前端只展示对应的下一步操作。

core/schemas/review_gate.py, core/agents/review_agent.py

4.2 前端功能模块

workbench

项目主工作台，承载多 Agent 对话、项目选单、工件查看、附件输入、流式输出、流程按钮和工具执行过程展示。

settings

配置中心，管理模型路由、工作区清单、引擎连接、工具权限、Skills 管理、用户画像和历史项目。

copywriting

文案加工台，挂在 Design Agent 子能力下，用于生成台词、物品描述、世界观片段，并导出 Markdown / CSV。

aigc

AIGC 能力中心，提供图像、声音、3D 模型、原型设计和文字创作网站的快捷入口，服务资产生产与灵感探索。

mcp

引擎连接页面，展示 Unity、Godot、Blender、Unreal 的配置状态、健康检查、工具映射和安装指引。

skills

项目级 Skill 使用入口与全局 Skill 管理入口，支持外部导入、启用/禁用和自我沉淀演示流程。

github

GitHub 协作视图，绑定仓库后只读展示 issue、PR、commit、CI 等信息，作为 PM Agent 的协作平台子能力。

game-model

游戏内模型接入的产品入口，用于规划 NPC 对话、敌人行为、动态任务提示等“AI 伴生”玩法能力。

4.3 功能中心与后端能力的关系

功能中心并不直接等同后端能力层。前端 feature 负责让用户看见、选择和确认能力；后端 capability 负责真正执行权限审核、工具调用、上下文解析和状态落盘。例如用户在 mcp 页面看到 Unity 已连接，真正的工具调用仍会进入后端 capabilities/mcp Adapter；用户在 copywriting 页面生成台词，实际 schema、上下文加载和 job 进度由后端文案能力完成。

5. 人机协作流程

这一节以“新人想做一个小型游戏 Demo”为例，说明 Ludens-Flow 如何作为辅助性产品参与全流程。

Step 1

输入粗略想法

用户只需要描述“想做一个校园塔防 / 像素冒险 / 对话解谜”等初始想法。Design Agent 会追问玩法核心、题材边界、美术成本和 MVP 范围，而不是直接输出大而全设定。

Step 2

生成并确认 GDD

系统将讨论结果写入 GDD.md。用户可以编辑，也可以导入已有策划案进入快速开发。GDD 定稿后才进入 PM 阶段。

Step 3

PM 拆解任务

PM Agent 读取 GDD，生成 PROJECT_PLAN.md，区分 MVP 必做项、延后项和风险。新人由此理解“先做什么”和“为什么先做”。

Step 4

Engineering 输出工程方案

Engineering Agent 根据目标引擎、项目工件和工作区结构生成 IMPLEMENTATION_PLAN.md。如果项目设置为 Godot，就注入 Godot 专用规则；如果没有设置，则使用通用小型游戏 demo 视角，不默认 Unity。

Step 5

Review Gate 评审

Review Agent 审查 GDD、计划和工程方案的一致性，给出 PASS / BLOCK / MAJOR 等问题。用户可以选择回流修正或进入持续开发。

Step 6

持续开发与工具执行

进入 DEV_COACHING 后，Agent 可以读取工作区、创建目录、写脚本、patch 文件、调用 Unity/Godot/Blender MCP。前端展示权限确认、执行中、失败或成功结果。

Step 7

生成项目文案与数据表

用户可用文案工作台生成 NPC 台词、物品描述、任务文本和世界观片段，并导出可进入游戏工程的 Markdown 或 CSV，例如 dialogue.csv、quest.csv、item.csv。

Step 8

生产美术与声音资产

用户从 AIGC 中心快速访问图像、声音、3D 模型和原型设计工具，例如生成角色立绘、敌人图标、UI 草图、环境音或低模资产；生成结果再作为附件或工作区文件交给 Agent 解释、整理和写入资源清单。

Step 9

写入工程并操作引擎

Engineering Agent 在用户批准的工作区内创建目录、写入脚本、Patch 文件，并通过 Unity / Godot / Blender MCP 执行受控操作，例如读取场景层级、创建对象、移动物体、保存场景或生成脚本骨架。

Step 10

接入游戏内模型能力

当项目需要“AI 原生”玩法时，工作台可以规划游戏内大模型调用方式：例如 NPC 对话、敌人行为决策、动态任务提示或教学反馈。PM Agent 控制功能范围，Engineering Agent 给出客户端调用、状态缓存、失败兜底和成本控制方案。

Step 11

测试、回流与协作追踪

用户运行 Demo 后把错误日志、截图或录屏作为附件交给系统。Agent 读取工作区文件和 Console 结果定位问题，Review Agent 再评估设计、工程和体验是否需要回流。GitHub 页面用于查看 issue、PR、commit 和 CI 状态。

Step 12

沉淀成可复用流程

当某类任务反复出现，例如“生成一关敌人配置并写入 CSV”或“从立绘生成 NPC 设定与台词”，用户可以开启 Skills 自我沉淀，把稳定流程保存为项目可启用 Skill，供后续游戏项目复用。

Step 13

形成 AI 伴生游戏项目

最终项目不只是“用 AI 生成过内容”，而是形成一套可继续迭代的游戏工程：有 GDD、计划、实现方案、代码、资产、数据表、引擎场景、协作记录，以及可在游戏内调用模型能力的玩法模块。

从交互角度看，Ludens-Flow 不是“给答案”，而是提供一个可解释的协作过程：用户能看到当前阶段、Agent 角色、工件、工具动作、权限确认和下一步选择。

6. 未来规划

产品与工程方向

  • 完善任务执行循环：读取上下文、计划、执行工具、观察结果、继续修正。
  • 强化 Patch / Diff 级编辑体验，展示将改什么、已改什么、风险是什么。
  • 继续推进 Unity、Godot、Blender、Unreal 的 MCP 接入与统一能力层。
  • 丰富 Skills 生态：外部导入、自我沉淀、项目级启用、适用阶段和工具权限。
  • 探索 Tauri 桌面版，降低本地启动和配置门槛。

AI 开发游戏 Benchmark

项目之外，希望建立专门衡量“AI 辅助开发游戏”的 Benchmark，而不是只用通用问答或代码题评估。计划重点覆盖需求澄清、工件一致性、工具调用质量、人机协作、异常恢复和真实产出等维度。

6.1 Benchmark 计划指标

指标评估问题与比赛维度的关系
阶段推进成功率能否从 GDD 到 PM 到 ENG 到 Review 稳定推进鲁棒性、效率
工件完整性生成的 GDD、计划、工程方案是否结构完整且互相一致实用价值
工具调用正确率是否正确选择文件、MCP、GitHub 或文案工具，并处理失败技术创新、鲁棒性
人类决策清晰度用户是否能理解当前状态和下一步选择人机交互
真实可用产出是否生成可进入游戏工程的代码、CSV、配置或引擎对象应用价值
Ludens-Flow 的核心观点是：游戏制作 Agent 不应该只是“生成游戏内容”的工具，而应该成为一个有边界、有状态、有工件、有工具能力的协作系统。
