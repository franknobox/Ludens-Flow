# Ludens-Flow 前端优先开发计划

> 目标：先定页面逻辑、信息架构、UI/UX 和状态反馈，再接真实后端功能。  
> 范围：只讨论 **ROADMAP 中尚未完成部分的纯前端工作**。

---

## 开发原则

- **产品能力前端先行**：先定入口、页面、状态和反馈，再做后端。
- **基础设施后端先行**：路径审核、schema、runtime 这类底层能力可以先后端验证。
- **先做可验证版本**：页面壳、假数据、状态流先跑通，再接真实接口。
- **先收口再扩展**：不在 UI 还没稳定时同时拉太多后端能力。

## 开发顺序

## P0. 工作台与设置页收口
- **目标**：统一 Workbench / Settings 的页面结构和状态心智。
- **完成情况**：基本完成，进入持续打磨。
- **前端重点**：
  - 已完成 ProjectToolbar、Settings sections、Workbench controller hook、MainPanel 子组件拆分。
  - 已完成工作台 / 设置页顶部项目状态共享、设置导航、工具页、工作区页、历史项目页基础节奏。
  - 后续只保留小范围 UI/状态一致性打磨。

## P1. 文件写入与权限确认流
- **目标**：把“写文件”做成可见、可确认、可回看的一条用户流程。
- **完成情况**：主体完成，待真实场景验证。
- **前端重点**：
  - 已完成工具执行过程卡、权限校验 / 授权 / 拒绝 / 文件变更事件展示。
  - 已完成写入前确认弹窗、项目级写入总开关、逐次确认与安全自动执行模式说明。
  - 已完成 workspace writable 可视化。
  - 后续重点是批量变更摘要、失败恢复和更清楚的“本轮改动文件”展示。

## P2. 工具能力中心
- **目标**：让工具页从“说明列表”升级为“能力中心”。
- **完成情况**：基础目录已完成，能力中心化未完成。
- **前端重点**：
  - 已完成 Settings 中工具目录与读写/工作区状态展示。
  - 未完成最近执行、最近失败、最近改动。
  - 未完成本轮参考文件 / 修改文件 / 使用工作区的稳定聚合视图。

## P3. Unity 工程能力页面化
- **目标**：为 Unity 文件读写、检索、MCP 和编辑器能力提供长期前端承载。
- **完成情况**：入口与 MCP 页面壳已完成，Unity 专项工程页未完成。
- **前端重点**：
  - 已完成 MCP 页面入口，覆盖 Unity / Godot / UE / Blender 等方向的入口表达。
  - 工作区清单已支持 Unity / Generic / Blender 类型。
  - 未完成 Unity 工程浏览专页与编辑器操作面板。
  - 高风险能力状态提示

## P4. 结构化结果承接
- **目标**：让前端开始稳定消费结构化结果，而不只显示纯文本。
- **完成情况**：部分完成。
- **前端重点**：
  - 已完成 Markdown 渲染，聊天回复和工件阅读体验明显改善。
  - 已完成工具过程事件的结构化展示。
  - 未完成通用结构化回复视图。
  - 未完成不同 Agent 的结构化承接样式。
  - 聊天区中的“文本 + 结构卡片”混合展示

## P5. 文案与表达子能力页面
- **目标**：把文案能力做成策划 Agent 的可配置子能力，而不是单独聊天框。
- **完成情况**：未开始。
- **前端重点**：
  - 文案任务定义页
  - 风格 preset / 用途 / 字数 / 限制配置
  - 结果页、版本比较、导出入口

## P6. 游戏内模型能力接入页
- **目标**：先定义“游戏内 Agent / Prompt / Tool 配置”在前端的完整路径。
- **完成情况**：页面壳与交互原型已完成，真实接入未完成。
- **前端重点**：
  - 已完成 Game Model 页面入口、模型卡、场景配置、Unity / REST 示例导出说明。
  - 后续重点是接真实配置保存、项目绑定和导出文件生成。

## P7. 更多引擎兼容页面壳
- **目标**：给 Godot / UE / Blender 等后续引擎兼容预留稳定入口。
- **完成情况**：基础入口已完成。
- **前端重点**：
  - 已通过 MCP 页面提供 Unity / Godot / UE / Blender 入口表达。
  - 未完成更完整的引擎兼容总览页和各引擎详情页。

## P8. 协作平台可视化
- **目标**：先定义 GitHub 等协作平台在工作台中的可视化呈现方式。
- **完成情况**：GitHub 可视化页面壳已完成，真实平台接入未完成。
- **前端重点**：
  - 已完成 GitHub 页面入口、分支 / 提交 / PR / Review / CI 的 mock 可视化。
  - 未完成真实 GitHub 数据接入。
  - 未完成与 Agent 工作流联动入口。

## P9. 外部 AIGC 集成页
- **目标**：把外部图片、声音、模型、原型设计能力做成统一入口。
- **完成情况**：快捷入口目录已完成，结果回流未完成。
- **前端重点**：
  - 已完成 AIGC 分类入口与主流平台快捷访问卡。
  - 已覆盖图片、视频、声音、3D、原型设计等分类。
  - 未完成结果回收、收藏、账号备注和资产组织流程。

## P10. Benchmark 与验收可视化
- **目标**：定义 benchmark、核心指标和 release checklist 的前端呈现。
- **完成情况**：未开始。
- **前端重点**：
  - 质量/评测页面
  - benchmark 结果卡
  - 指标面板
  - release checklist 页壳

## 推荐推进顺序

### 第一阶段：近期收口
1. `P1` 文件写入与权限确认流：继续补批量变更摘要、失败恢复和本轮改动文件展示。
2. `P2` 工具能力中心：补最近执行 / 最近失败 / 最近改动。
3. `P4` 结构化结果承接：补通用结构化回复视图和 Agent 差异化卡片。
4. `P10` Benchmark 与验收可视化：补收版前质量页与 Release Checklist。

### 第二阶段：中期差异化能力
5. `P3` Unity 工程能力页面化：补 Unity 工程浏览与编辑器操作面板。
6. `P5` 文案与表达子能力页面。
7. `P6` 游戏内模型能力接入页：从页面壳推进到真实配置保存和导出。

### 第三阶段：平台化方向预留
8. `P7` 更多引擎兼容页面壳：从 MCP 入口扩展为完整总览。
9. `P8` 协作平台可视化：接真实 GitHub 数据。
10. `P9` 外部 AIGC 集成页：补收藏、结果回流和资产组织。

---

## 前端阶段的交付要求

每个功能在前端阶段至少要交付：

- 页面入口
- 信息架构
- 核心交互路径
- loading / empty / error / success
- destructive action confirmation
- mock 数据
- 状态切换逻辑
- 必要的接口契约草案
